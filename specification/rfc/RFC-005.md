# RFC-005 — Approved Technology Registry

## Purpose

This RFC is the canonical technology allowlist for Executive Command Center. No engineer or AI coding agent may introduce a framework, library, database, protocol, runtime or infrastructure component unless it is listed here for the active phase or approved through an ADR.

## Interpretation rules

1. **Required now** means the technology is part of the current Phase 0 implementation baseline.
2. **Approved later** means it may be used only when a future phase specification activates it.
3. **Evaluation only** means it is not approved for implementation.
4. Exact direct-dependency versions are recorded below and in lockfiles.
5. Transitive dependencies are controlled exclusively through `uv.lock` and `pnpm-lock.yaml`.
6. Floating versions such as `latest`, `3.x`, `5.x`, `^` and `~` are prohibited in manifests, container tags and CI actions.

## Phase 0 architecture baseline

```text
Browser
  ↓
React web application
  ↓
FastAPI modular monolith
  ↓
PostgreSQL 18
  ├── relational domain data
  ├── PKOS nodes, edges and provenance
  ├── durable event outbox/inbox
  └── session and workspace state
```

Phase 0 deliberately excludes Neo4j, Qdrant, Redis, NATS, Kafka, Kubernetes and cloud-managed services.

| PKOS graph representation | PostgreSQL node and edge tables |
| Vector search | Deferred |

| Authentication | Local owner account with server-side opaque session |

## API and contract standards

- REST over HTTP
- OpenAPI 3.1
- JSON payloads
- UUID identifiers
- UTC timestamps in RFC 3339 form
- typed command, query and event contracts
- no GraphQL or gRPC in Phase 0

## Authentication baseline

Phase 0 uses:

- one local owner account per installation
- Argon2id password hashing
- server-side opaque sessions persisted in PostgreSQL
- `HttpOnly`, `Secure` outside local HTTP development, and `SameSite=Lax` cookies
- CSRF token validation for state-changing browser requests
- 12-hour idle timeout and 7-day absolute session lifetime
- session rotation after authentication and privilege-sensitive operations

JWT is not approved for local browser sessions. OAuth2 is reserved for external connector authorization and future multi-user identity.

CI MUST fail on verified committed secrets and critical vulnerabilities without an approved, time-bound exception.

## Observability baseline

Phase 0 uses structured JSON logs and application-native metrics endpoints. OpenTelemetry, Prometheus and Grafana are approved later and are not required local services in Phase 0.

Required log fields:

- timestamp
- level
- service
- domain
- workspace_id where applicable
- correlation_id
- causation_id where applicable
- request_id
- event_id where applicable
- message

Secrets, credentials, full prompt context and sensitive source content MUST NOT be logged.

## Version and lock policy

- Direct dependencies use exact equality pins.
- Python uses `requires-python == 3.14.*` plus a checked-in `.python-version` containing `3.14.6`.
- Node uses `.node-version` containing `22.17.0` and an exact `packageManager` field for pnpm.
- Docker images use immutable version tags and SHOULD record image digests after initial resolution.
- GitHub Actions MUST use immutable commit SHAs, with the human-readable release in a comment.
- `uv.lock` and `pnpm-lock.yaml` are mandatory and committed.
- Lockfile changes require CI, vulnerability scans and review.

## Upgrade policy

Patch and minor upgrades require:

1. compatibility tests
2. security scan
3. lockfile diff review
4. changelog review
5. rollback instructions

Major runtime, framework, database or infrastructure upgrades require an ADR.

## Prohibited in Phase 0

- floating dependency versions
- unpinned Docker images
- Neo4j
- Qdrant
- Redis
- Kafka
- NATS
- Temporal
- Kubernetes
- cloud-only dependencies
- JWT browser sessions
- no-auth development mode
- LangChain, LangGraph, CrewAI or AutoGen
- MongoDB, Firebase, Django, Flask, Express or Electron

## Fitness functions

- No dependency exists outside this registry or an accepted ADR.
- No floating version exists in manifests, workflows or image tags.
- Phase 0 Compose starts PostgreSQL only as shared infrastructure.
- Lockfiles reproduce on macOS and Linux.
- CI emits an SBOM and runs secret and vulnerability scans.
- Authentication tests prove cookie flags, CSRF protection, expiration and session rotation.
- No future-phase technology is required for local startup.

## Approval

RFC-005 version 1.1.0 is the approved technology baseline for Phase 0. Any deviation requires an ADR or an explicit versioned amendment to this RFC.
