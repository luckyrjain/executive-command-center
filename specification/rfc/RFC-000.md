# RFC-000 — Document Control

## Purpose

This RFC defines the complete documentation governance model for Executive Command Center.

Its purpose is stability. Documentation exists to direct delivery, not to create an endless architecture process.

## Governing rule

> Stability is more valuable than optimization.

Once a batch, structure or document is frozen, no contributor or AI agent may redesign it unless the user explicitly requests the change.

## Authority

Authority descends in this order:

1. explicit user instruction
2. accepted RFC
3. accepted ADR
4. active STD
5. approved SPEC
6. implementation and tests

When documents conflict, the higher authority wins. The conflict must be corrected before new implementation depending on it proceeds.

## Frozen project structure

```text
executive-command-center/
├── README.md
├── specification/
│   ├── rfc/
│   ├── adr/
│   ├── standards/
│   ├── phases/
│   ├── templates/
│   ├── schemas/
│   ├── prompts/
│   ├── diagrams/
│   └── wireframes/
├── backend/
├── frontend/
├── infra/
├── tools/
└── scripts/
```

This is frozen at v1.0.

Runtime manifests, tests, lockfiles and CI files may remain where their tooling requires. They are implementation artifacts, not specification document types.

## Allowed document types

Only four authoritative document types exist.

### RFC

Defines product direction, engineering philosophy, system architecture or another broad cross-cutting decision.

Location: `specification/rfc/`

Naming: `RFC-NNN.md`

### ADR

Records one material architecture decision, its context, alternatives and consequences.

Location: `specification/adr/`

Naming:

- proposed: `ADR-Proposed-NNN.md`
- accepted numbering may be assigned during review

An ADR does not silently override an RFC. The affected RFC must be versioned when an ADR changes its contract.

### STD

Defines mandatory engineering, repository, security, testing or operational rules.

Location: `specification/standards/`

Naming: `STD-NNN.md`

### SPEC

Defines an implementable phase or capability: scope, requirements, data, API, UX, security, tests, acceptance and rollback.

Location: `specification/phases/`

Naming: `SPEC-NNN-short-name.md`

## Supporting artifacts

The following directories may contain artifacts referenced by an RFC, ADR, STD or SPEC:

- `specification/templates/`
- `specification/schemas/`
- `specification/prompts/`
- `specification/diagrams/`
- `specification/wireframes/`

These artifacts are not independent document types. They cannot define policy or scope without an authoritative parent document.

Independent runbooks, review reports, status documents, roadmaps, contract documents and process notes are prohibited. Required content belongs in the governing RFC, ADR, STD or SPEC.

## Frozen Batch 1

Batch 1 contains exactly:

1. `README.md`
2. `RFC-001 Product Definition`
3. `RFC-002 Engineering Philosophy`
4. `RFC-003 Design Principles`
5. `RFC-004 System Architecture`
6. `RFC-005 Technology Registry`
7. `STD-001 Repository Standards`

Nothing may be added or removed without explicit user instruction.

## Document lifecycle

Allowed statuses:

- Draft
- In Review
- Accepted
- Superseded
- Rejected

Only Accepted RFCs and ADRs, active STDs and approved SPECs authorize implementation.

Draft content may guide discussion but is not an implementation contract.

## Required metadata

Every authoritative document contains:

```yaml
id: RFC-001
title: Product Definition
status: Draft
version: 1.0.0
owner: Lucky Jain
```

Dependencies and reviewers are included when relevant.

## Versioning

Documents use semantic versioning:

- patch: clarification with no behavioral change
- minor: compatible addition or materially expanded guidance
- major: breaking change to scope, behavior, ownership or architecture

Accepted content is never silently rewritten. Every change updates the version and changelog.

## Frozen writing process

Every document follows exactly:

```text
Write
  -> Review
  -> Revise
  -> Commit
  -> Next document
```

One document is completed before the next document begins.

Process redesign is not a delivery artifact and must not consume a response intended for document production.

## Review requirements

Review checks:

- purpose and scope are explicit
- in-scope and out-of-scope boundaries do not conflict
- normative language is testable
- dependencies exist or are clearly future
- security and privacy implications are stated
- architecture aligns with RFC-004
- technology aligns with RFC-005
- repository rules align with STD-001
- links and identifiers resolve
- no unauthorized document type is introduced
- no speculative future work is presented as approved

Blocking findings are Critical, High or Medium. A document is not accepted while any blocking finding remains unresolved.

## Revision rule

Revision corrects review findings in the active document. It does not redesign the repository or expand the batch.

If a better architectural idea appears:

1. do not change the active plan
2. finish the current document
3. record the idea later as `ADR-Proposed-XXX`
4. continue with the frozen specification

## Commit rule

Each Batch 1 document receives an intentional commit after review and revision.

Commit messages are concise and identify the document. Unrelated implementation changes are not mixed into a document commit.

Structural cleanup required to enforce the frozen batch may be committed once after all eight documents are in place.

## Change control

Only the user may authorize:

- a new document type
- a new Batch 1 document
- removal of a Batch 1 document
- a different top-level logical directory
- a different writing process
- reopening a frozen architecture decision without an ADR

An explicit user request must identify the intended change. Approval for one change does not authorize adjacent redesign.

## Stop rule

Stop and ask the user only when:

- two instructions materially conflict
- a destructive change has an unclear target
- required authority is missing
- the choice changes product scope or architecture

Otherwise, make the smallest safe assumption and deliver the active artifact.

## Traceability

Implementation changes cite the governing RFC, ADR, STD or SPEC in the pull request.

Requirements use stable identifiers when needed. A requirement is not duplicated across documents; the dependent document references its owner.

Code does not retroactively define the specification. If implementation and specification differ, either the implementation is corrected or the authoritative document is explicitly revised.

## Quality gates

Before committing a document:

- front matter is valid
- headings are well formed
- internal links resolve
- no prohibited document type is referenced as authoritative
- terminology and identifiers are consistent
- Markdown validation passes
- the document has been reviewed for contradictions and omissions

Before merging a documentation batch:

- the exact batch file list matches the frozen list
- the repository structure matches v1.0
- deleted documents have no remaining authoritative references
- implementation tests still pass when documentation movement can affect tooling
- the pull request has no blocking review findings

## AI contributor rules

An AI contributor:

- follows the frozen sequence
- produces the requested artifact instead of redesigning process
- does not add helpful-but-unrequested documents
- does not promote a Draft
- does not invent user approval
- does not broaden destructive scope
- records future architecture ideas as proposed ADRs only after the active artifact
- reports validation honestly

## Definition of complete

A document is complete when:

- it exists at the frozen path
- it uses an allowed type
- required metadata is present
- its scope and requirements are internally consistent
- review findings are resolved
- Markdown and links validate
- it is committed intentionally

Batch 1 is complete when all eight documents satisfy this definition and no conflicting authoritative document remains.
