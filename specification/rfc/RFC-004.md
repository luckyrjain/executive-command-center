# RFC-004 — System Architecture

## Purpose

This RFC is the single authoritative architecture document for Executive Command Center. It replaces the earlier chapter-based architecture set.

ECC is a local-first modular monolith. The architecture prioritizes understandable boundaries, deterministic operation, user-owned data, explicit evidence and human authority over premature distribution or autonomous behavior.

## Architectural goals

- Run the complete core product on one developer machine.
- Keep business state authoritative outside AI models.
- Make domain ownership and dependency direction explicit.
- Preserve offline and AI-disabled operation.
- Support incremental capability delivery without speculative infrastructure.
- Make every recommendation traceable to evidence.
- Require durable confirmation before consequential state changes.
- Keep infrastructure replaceable through narrow ports.

## Non-goals

- Microservices before measured scaling need.
- Mandatory cloud infrastructure or cloud AI.
- Direct cross-domain database access.
- Provider SDKs inside domain logic.
- Autonomous agents owning business state.
- Dedicated graph, vector, cache or message infrastructure in the initial baseline.
- Multi-user or enterprise controls before an approved SPEC activates them.

## Runtime topology

The initial supported topology contains:

- one React frontend
- one FastAPI backend process
- one PostgreSQL instance
- background work executed through bounded in-process workers backed by durable PostgreSQL state

New runtime services require:

1. a measured limitation in the current topology
2. an ADR
3. an RFC-005 update
4. an approved SPEC activating the technology

## Backend architecture

The backend is a modular monolith with three logical layers.

### Domain

Owns entities, invariants, state transitions, domain services and domain events. Domain code does not depend on FastAPI, SQLAlchemy, provider SDKs or external transports.

### Application

Owns commands, queries, use-case orchestration, authorization checks and transaction boundaries. It depends on domain code and declared ports.

### Infrastructure

Implements persistence, HTTP, sessions, event delivery, model providers and external connectors. Infrastructure may depend inward; domain and application code may not depend outward.

### Dependency rule

```text
Interface/Infrastructure -> Application -> Domain
```

Cross-domain collaboration uses typed application contracts or events. A domain never queries another domain's tables directly.

## Core domains

The target system may contain the following bounded domains when activated by an approved SPEC:

- identity and workspace
- tasks and commitments
- notes and knowledge
- calendar and meetings
- risks and attention
- recommendations and planning
- audit and evidence
- connectors and synchronization
- automation

The active SPEC determines which domains may be implemented. This RFC does not activate future scope.

## Data architecture

PostgreSQL is the authoritative store.

Every authoritative record includes:

- stable identifier
- workspace identifier
- lifecycle state
- created and updated timestamps
- optimistic version where mutable

Rules:

- foreign keys include the workspace boundary where applicable
- cross-workspace identifiers return not found
- migrations are version controlled
- destructive migration requires an explicit recovery plan
- derived search, ranking, timeline, summary and embedding data is rebuildable
- source deletion and permission loss propagate to derived data

Dedicated graph or vector storage is not approved by this RFC. It requires an ADR and technology-registry update.

## Event architecture

Domain events use a versioned envelope containing:

- event identifier
- event type and schema version
- aggregate identifier and version
- workspace identifier
- occurred timestamp
- correlation and causation identifiers
- redacted payload

The initial event mechanism is an in-process publisher with PostgreSQL outbox/inbox durability.

Delivery semantics:

- at least once
- idempotent consumers
- bounded retry
- dead-letter state
- replay with authorization and schema checks

Distributed messaging is deferred until measured throughput or isolation needs justify it.

## API architecture

HTTP APIs are versioned under `/api/v1`.

Conventions:

- JSON request and response contracts
- opaque server-side browser sessions
- server-derived actor and workspace
- problem-details error envelope
- idempotency keys for creates and actions
- optimistic concurrency for mutable records
- signed cursors for pagination
- UTC storage with workspace-timezone presentation
- OpenAPI generated and validated in CI

The browser may not assert actor, workspace, tenant or accountable owner.

## Frontend architecture

The frontend is a React application organized by feature.

Rules:

- server state uses the approved query library
- local UI state remains separate from authoritative server state
- business invariants remain on the backend
- feature modules do not import private internals from other features
- every primary view implements loading, empty, degraded, offline, error and conflict states
- core flows meet WCAG 2.2 AA
- deterministic functionality remains usable when AI is unavailable

## Knowledge and evidence

ECC separates authoritative facts from generated summaries.

Every knowledge claim or recommendation must retain:

- source reference
- observed/effective time
- evidence availability
- confidence
- generating policy/model version when applicable

Summaries and embeddings are disposable projections. Evidence and authoritative records are not.

## AI runtime boundary

All model use passes through a typed model-runtime port.

Each invocation declares:

- task type
- data classification
- permitted provider class
- input and output schema
- timeout, token, cost and step budget
- fallback behavior

AI output never directly owns business state. Invalid structured output is discarded. Consequential actions require the domain's confirmation contract.

Remote model use is opt-in and policy controlled. Retrieved content and tool output are treated as untrusted data.

## Connector boundary

Connectors translate external provider data into normalized application commands and projections.

Rules:

- source systems remain authoritative
- credentials use least privilege and opaque secret references
- synchronization is incremental, idempotent and resumable
- webhook and polling overlap is deduplicated
- provider payloads are untrusted
- connectors never write directly to domain tables
- permission loss and deletion propagate locally
- external write actions require approved automation authority

## Attention and recommendation boundary

Attention ranks work, not people.

Every ranked item exposes factors, evidence, confidence, freshness and policy version. Protected characteristics, personality inference, activity volume and response-speed performance proxies are prohibited.

Recommendations progress from proposal to explicit confirmation before execution. The system does not silently perform consequential executive actions.

## Security architecture

Security is enforced at every boundary:

- Argon2id password hashing
- opaque HttpOnly sessions
- CSRF protection
- secure cookie attributes
- workspace isolation in queries and constraints
- input and output validation
- redacted audit
- secret scanning and dependency scanning
- provider egress controls
- least-privilege connector credentials
- prompt-injection and tool allowlist controls

Logs, metrics and traces must not contain secrets, note bodies, raw sensitive prompts or private evidence.

## Observability

Every request and durable event carries correlation identifiers.

Required signals:

- request rate, duration and error
- database pool and query health
- migration and readiness state
- event publish, retry and dead-letter state
- background-job queue age and outcomes
- model invocation status, latency and bounded usage
- connector freshness and failure
- authorization denial reason codes

Telemetry cardinality is bounded and private content is excluded.

## Reliability and recovery

- readiness verifies required dependencies
- graceful shutdown preserves in-flight transaction safety
- retries are limited to classified transient failures
- unknown external outcomes require review
- backups are checksummed and restored in a clean environment
- derived projections can be rebuilt from authoritative records
- feature flags disable optional capabilities without corrupting data

## Deployment architecture

Local development is the required baseline. Container packaging may be used for reproducibility.

Cloud, Kubernetes, multi-region, multi-tenant and enterprise deployment profiles are deferred until activated by an approved SPEC and ADR. No future deployment assumption may force cloud dependency into the core domain.

## Architecture fitness functions

CI must verify:

- forbidden dependency directions
- no cross-domain persistence imports
- technology allowlist compliance
- generated API contract consistency
- migration upgrade and recovery
- workspace-isolation coverage
- event idempotency
- deterministic AI-disabled core flows
- vulnerability, secret and container scans
- Markdown link validity for Batch 1

## Change control

Material changes to domain boundaries, data ownership, runtime topology, security, external side effects or approved technologies require an ADR.

An architecture idea discovered during active document delivery does not interrupt the frozen batch. It is recorded later as `ADR-Proposed-XXX`.

## Acceptance criteria

RFC-004 is ready for acceptance when:

- the architecture fits the frozen project structure
- RFC-005 contains every active technology
- STD-001 enforces the dependency and repository rules
- no required rule depends on a deleted architecture chapter
- core operation is possible without cloud AI
- state-changing AI or automation requires explicit authority
- security, observability and recovery boundaries are testable
