# STD-001 — Repository Standards

---

# Executive Summary

This standard defines how software is built inside the Executive Command Center repository.

Unlike RFCs, which define architecture and intent, this document defines implementation rules.

Every engineer.

Every AI coding agent.

Every pull request.

Must comply with these standards.

Violation of these standards is considered a defect.

---

# Guiding Principles

Repository standards exist to ensure the codebase remains:

- Understandable
- Predictable
- Testable
- Reviewable
- AI-friendly
- Long-lived

Optimization for short-term development speed is explicitly rejected if it harms long-term maintainability.

---

# Repository Structure

The authoritative logical structure is frozen at version 1.0.

```text
executive-command-center/
├── README.md
├── specification/
│   ├── rfc/
│   ├── adr/
│   ├── standards/
│   ├── phases/
│   ├── templates/
│   ├── schemas/
│   ├── prompts/
│   ├── diagrams/
│   └── wireframes/
├── backend/
├── frontend/
├── infra/
├── tools/
└── scripts/
```

No new logical top-level directory is permitted unless the user requests it explicitly.

Runtime manifests, lockfiles, tests, CI configuration and editor configuration MAY remain at conventional implementation locations. They do not create specification document types or change the logical architecture.

The former `docs/` tree is prohibited.

# Documentation Rule

Only RFC, ADR, STD and SPEC are authoritative document types.

Supporting schemas, prompts, diagrams and wireframes MUST be referenced by one of those document types and MUST NOT define independent policy.

Behaviour changes require code, tests and the affected authoritative document in the same pull request.

Specification drift is prohibited.

---

# Feature Development Workflow

```text
Write
  -> Review
  -> Revise
  -> Commit
  -> Next document
```

For implementation:

```text
Approved RFC/ADR/STD/SPEC
  -> Implement
  -> Test
  -> Review
  -> Merge
```

An improvement idea discovered during active delivery MUST NOT redesign the current plan. Record it later as `ADR-Proposed-XXX` and continue unchanged.

---

# Stop-and-Ask Protocol

If any requirement is

Ambiguous

Missing

Contradictory

Incomplete

The developer MUST stop.

The developer MUST ask the user when ambiguity materially changes scope. If work can continue safely, record the architectural alternative later as `ADR-Proposed-XXX`.

Never guess.

# Repository Oath

Every contributor agrees to leave the repository in a better state than they found it.

Before merging ask:

- Is it simpler?
- Is it clearer?
- Is it more maintainable?
- Is it better documented?
- Does it align with the specification?

If the answer is no,

the change is not ready.

# Summary

STD-001 is the operational constitution of the Executive Command Center repository.

It ensures that every contribution—whether written by a human or generated by AI—is consistent with the architectural vision, technology registry, engineering philosophy, and long-term maintainability goals of the platform.

This document is intentionally prescriptive.

Consistency is considered a competitive advantage.
