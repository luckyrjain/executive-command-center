# Phase 4 AI Runtime Design

**Status of this document:** planning artifact only. It does not by itself change `docs/phases/PHASE-004-ai-runtime.md`'s status, does not start implementation, and does not close Phase 3's open exit gate (the two-week dogfood window, `docs/runbooks/PHASE-3-DOGFOOD.md`, is still at 0/14 days). Per `docs/ROADMAP.md`'s approval gates, Phase 4 implementation may not begin until: this document's decisions are reviewed and accepted by the repository owner (or the repository owner grants the same kind of parallel-start exception Phase 2 and Phase 3 each received), `docs/phases/phase-004/*.md` contracts move from Draft to Approved for Implementation, and the four approval-gate items `docs/phases/PHASE-REVIEW.md:135` names for Phase 4 (approved local/remote models and providers, data-class egress matrix, evaluation floors, trace retention) are resolved. This document exists to make that decision informed, and to leave an implementation-ready plan queued for the moment it's granted.

## Outcome

Activate Ollama as ECC's local model-inference technology (pre-registered in `RFC-005.md`'s "Approved later" table since Phase 0, gated on exactly "AI-runtime phase specification and ADR review" — this document and the paired ADR are that gate) and design the first, deliberately narrow slice of the AI Runtime `ADR-0004`/`ADR-0007` already committed this repository to: a provider-neutral Model Router with one local provider registered, versioned prompts/tools, schema-enforced structured output, a bounded read-only tool runtime, and an evaluation harness with a real first dataset. Scope is exactly `docs/phases/PHASE-004-ai-runtime.md`; this document does not expand it, and does not touch Phase 5's automation/action semantics.

## Why this isn't a green field

Unlike Phase 2 (PKOS) and Phase 3 (`attention_items`), there is no already-shipped table or module this phase must reconcile against — Phase 4 has no code yet. But it is not a blank page either:

- `ADR-0004-ai-runtime.md` (Accepted) already commits this repository to a single AI Runtime, provider-neutral Model Router, prompt registry, structured-output validation, tool permission checks, evaluation hooks and audit logging, with domain services never calling providers directly and AI outputs remaining proposals until a human or deterministic domain logic confirms them. This document does not redecide any of that — it makes it concrete.
- `ADR-0007-model-router.md` (Accepted) already commits to policy-based routing on task type, data sensitivity, capability, context size, latency budget and provider availability; local preferred for sensitive content; fallback explicit and auditable; sensitive requests never silently fall back to cloud. This document's routing algorithm (Decision 2 below) is the concrete first-cut implementation of that policy, not a new one.
- `RFC-005.md`'s "Approved later" table pre-registered Ollama in Phase 0 with an explicit two-part activation gate: an AI-runtime phase specification and an ADR. `PHASE-004-ai-runtime.md` already exists as the phase specification; this document plus the paired ADR (`docs/adr/ADR-0012-ollama-local-inference.md`) is the second half of that gate — the same two-artifact pattern Phase 2's `pgvector`/`sentence-transformers` activation used (design work + ADR-0011 + RFC-005 v1.2.0 amendment).
- `docs/architecture/chapter-03-ai-runtime.md` (RFC-004 Chapter 3, still Draft) describes a considerably larger aspirational architecture — multi-agent orchestration, a Reflection Engine, an Agent Runtime with named agents ("Morning Brief Agent," "Meeting Preparation Agent"), and a multi-model routing table naming Qwen Coder, DeepSeek and Nomic alongside Qwen Instruct. `PHASE-004-ai-runtime.md`'s own "Out of scope" (`open-ended background agents`) and "Deferred backlog" (`unbounded agents`) are narrower than that chapter by design — this document follows the phase specification, not the aspirational chapter, and treats the chapter as directional context for later phases (Agent Runtime and multi-model routing are explicitly deferred below, not built here). Where the two agree — a Model Router as the only path to a model, tool contracts with input/output schema and audit, prompts owned by the runtime rather than domains, "Domains SHALL NOT call LLMs directly" (`ARC-AI-001`) — this design conforms exactly, including reusing the chapter's own "Planning → Qwen Instruct" mapping (line 206) as supporting precedent for Decision 1's model family choice.
- Phase 3's `meeting_prep.py` already anticipated this phase: its optional AI-enrichment section is feature-flagged off "until Phase 4 exists to serve it" (`docs/superpowers/specs/2026-07-22-phase-3-human-attention-engine-design.md`'s Meeting preparation approach). This document does not wire that flag on — see Decision 8 (deferred scope) — but the runtime designed here is shaped so that flipping it is a Phase-4-consumer integration, not a redesign.

## Decision 1: model runtime and first local model

**Model runtime: Ollama**, as pre-registered. No alternative runtime was independently evaluated for this activation — RFC-005 already named Ollama as the specific technology whose gate this document satisfies, and `ADR-0004`/chapter-03 already assume an Ollama-shaped local inference boundary (`AFF-AI-001`: "No direct Ollama calls outside Model Router"). The paired ADR's Alternatives considered section evaluates llama.cpp and ONNX Runtime as the genuine substitutes and records why Ollama is still preferred for this activation specifically.

**First model: `qwen2.5:1.5b-instruct-q4_K_M`** (Qwen2.5, 1.5B parameters, Q4_K_M quantization, served through Ollama). Concrete reasoning, mirroring how `ADR-0011` named `sentence-transformers/all-MiniLM-L6-v2` specifically rather than leaving embeddings generic:

- **Size and footprint.** At Q4_K_M quantization a 1.5B model needs roughly 1-1.5 GB of resident memory and runs at usable tokens/second on CPU-only hardware — consistent with ADR-0002's local-first architecture and this being a single-machine, resource-constrained deployment (the same class of machine already running PostgreSQL, FastAPI and the frontend build in Docker Compose). A 7B+ model would make "runs on the executive's own laptop with no GPU" an aspiration rather than a fact for this first activation.
- **Instruction-following and structured output.** Qwen2.5's instruct-tuned checkpoints are specifically trained for JSON/structured-output adherence and tool-call formatting, which matters directly for Decision 4 (schema validation is the enforcement backstop, but a model that already tends to emit well-formed JSON means fewer `schema_invalid` failures and retries in practice).
- **Consistency with existing direction.** `chapter-03-ai-runtime.md`'s own model-selection table (line 206) already names "Qwen Instruct" for Planning-shaped tasks — this activation picks a concrete, small member of the same family rather than introducing an unrelated one, keeping the eventual multi-model registry (deferred, Decision 8) coherent with that chapter's direction instead of contradicting it.
- **License and availability.** Qwen2.5 ships under Apache 2.0 (no field-of-use restriction relevant to this repository) and is a standard tag in Ollama's public model library, requiring no custom GGUF conversion or private hosting step.

**Alternatives considered and deferred, not rejected outright** (a second registered model is explicitly in scope for a later Phase 4 slice, not this one):
- `llama3.2:3b-instruct-q4_K_M` — roughly double the memory/latency cost for modest quality gain on the short, bounded, single-turn tasks Decision 6 scopes for slice 1 (attention-item explanation, grounded entity summarization). Registering it now would mean the routing algorithm (Decision 2) has to prove tie-breaking between two candidates before a single candidate's path is even validated end to end — deferred to the slice that actually needs the extra quality/context headroom.
- A 7B-class model (`qwen2.5:7b-instruct-q4_K_M` or similar) — rejected for this activation on the same CPU-latency-predictability grounds Decision 5's budget numbers below assume; revisit once real latency measurements exist from the 1.5B model in production use.

The registry (Decision 2) stores `model_id` as an opaque string (`ollama:qwen2.5:1.5b-instruct-q4_K_M`), so adding a second model or a second runtime later is a `model_definitions` row, not a schema change.

## Decision 2: model/provider registry shape and routing algorithm

`MODEL-ROUTING-CONTRACT.md` already names the eligibility dimensions (data residency/privacy, required capability, structured-output support, context limit, health, latency, budget) without an ordering. This document fixes the order and the tie-break, matching the contract's existing rule that routing must be "deterministic for a task, policy version, availability snapshot and data class":

**Registry (`model_definitions`, per `DATA-MODEL.md`):** each row declares `provider` (`ollama` for this activation), `model_id`, `deployment` (`local`), `data_classes` (the data classes this deployment is eligible to process — see Decision 7), `capabilities` (a small fixed set: `extraction`, `summarization`, `explanation` for slice 1 — not an open string), `context_window_tokens`, `structured_output_supported` (boolean; true here because Decision 4's validator is runtime-side and does not depend on the model's native mode), `status` (`active`/`disabled`).

**Eligibility pipeline** (hard filters, evaluated in this fixed order — a candidate failing any step is excluded, not deprioritized):

1. **Data residency/privacy.** The run's declared `data_class` must be in the candidate's `data_classes`. For this activation every registered model is `deployment=local`, and Decision 7 keeps every data class local-eligible, so this step never excludes the sole candidate today — but it stays first because it is the one filter that must never be bypassed by a later step's scoring, matching `ADR-0007`'s "sensitive requests may not silently fall back to cloud."
2. **Required capability.** The task's declared capability (from its typed port, never the caller) must be in the candidate's `capabilities`.
3. **Structured-output support.** If the task requires schema-validated output (Decision 4), the candidate must have `structured_output_supported=true`.
4. **Context limit.** `estimated_prompt_tokens + declared_max_output_tokens` (both computed before the call, not measured after) must fit within the candidate's `context_window_tokens` at a 10% safety margin (i.e. must fit within 90% of the window) — the margin absorbs tokenizer-estimate drift between ECC's estimate and Ollama's actual tokenizer.
5. **Health.** The candidate's circuit-breaker state (Decision 5) must not be `open`.
6. **Latency.** The candidate's rolling observed p95 latency for this task type must fit within the task's declared timeout minus a fixed 500ms overhead reserve (routing + validation + audit write).
7. **Budget.** Remaining run/session token and time budget (Decision 5) must be non-zero.

**Preference among remaining eligible candidates** (only reached when eligibility leaves more than one candidate — a no-op for slice 1's single-model registry, specified now so it does not have to be re-derived when a second model is added):

1. Local before remote (per `ADR-0007`; moot while no remote provider is registered — Decision 7).
2. Must meet the task's configured evaluation quality floor (Decision 6) — a candidate that has not passed evaluation for this task type is not preferred over one that has, even if otherwise eligible.
3. Lower expected cost (all `$0` for local-only in this activation; the field is scored anyway so the ordering is stable once remote candidates exist).
4. Lower observed p95 latency.
5. **Deterministic final tie-break:** ascending `model_id` string comparison. This is arbitrary but fixed, which is the only property `MODEL-ROUTING-CONTRACT.md`'s determinism requirement actually needs.

Routing overhead (this whole pipeline, excluding the model call itself) is pure in-memory comparison against a cached registry/circuit-state snapshot refreshed on a short interval (health/circuit state is never queried synchronously per-request) — keeping it inside the phase's own p95 <50ms non-functional requirement (Decision 5 gives a benchmark target).

## Decision 3: prompt/tool versioning mechanism

Concrete mechanism for `DATA-MODEL.md`'s `prompt_versions`/`tool_definitions` records:

- **Identity and hashing.** `prompt_id` is a stable slug (e.g. `attention.explain_item.v1` — the version is also embedded in the id for human readability, but `version` is the authoritative integer column). `template_hash` is `sha256` over the canonical (UTF-8, sorted-keys-where-applicable) bytes of `{template, input_schema_ref, output_schema_ref}`. `tool_definitions` uses the same `sha256` scheme over `{input_schema, output_schema, scopes, handler_ref}`.
- **Immutability.** Once a `prompt_versions`/`tool_definitions` row's `status` leaves `draft` (i.e. becomes `active` or `retired`), its `template`/`template_hash` (or schema/scope columns for tools) are immutable — enforced by a PostgreSQL trigger that rejects any `UPDATE` touching those columns once `status <> 'draft'`, not just an application-layer check, so a bug in the runtime code cannot silently mutate a version already referenced by a completed `ai_run`. Editing a prompt or tool always means inserting a new row with `version = previous + 1`; there is no in-place edit path at any layer.
- **Activation.** Exactly one `active` version per `prompt_id`/tool `name` at a time, enforced by a partial unique index (`WHERE status = 'active'`). Activating a new version is an explicit administrative action (`POST /ai/policies` scope in `API-SCHEMAS.md`'s administrative surface) that writes an audit event and does not retroactively change which version any already-completed `ai_run` recorded — every `ai_run` row pins the exact `prompt_version`/`tool_definitions.version` it used, so evaluation reproducibility (Decision 6) never depends on "whatever is active now."
- **No new dependency.** Hashing uses the standard library `hashlib.sha256`; the trigger is plain PostgreSQL DDL, matching every other immutability rule already enforced this way in the codebase (e.g. Phase 2's claim-supersede pattern).

## Decision 4: structured output validation

**Pydantic (already RFC-005's Phase 0 baseline, 2.11.7 — no new dependency)** is the structured-output enforcement mechanism, closing the phase's own hard acceptance criterion ("Structured invalid output cannot reach domain mutations"):

- Each task/tool's output contract is a Pydantic model; `output_schema_ref` in `prompt_versions`/`tool_definitions` points at that model's `model_json_schema()` output, stored as the versioned schema artifact (so the schema itself is hashed and immutable exactly like the prompt template).
- The AI Runtime validates every model response with `TypeAdapter(OutputModel).validate_json(raw_response)` (Pydantic's `strict` mode: no silent type coercion of the kind that would let a model's loosely-typed JSON slide through) before the response is visible to any caller. A validation failure never reaches the domain layer — the `ai_run` is marked `failed` with a classified `schema_invalid` error and the raw output is retained only in the (default-off, redacted) trace store, never returned as a usable result.
- **One bounded repair retry** is allowed on `schema_invalid` specifically (not on business-rule or safety failures): the runtime re-prompts once with the validation error appended as an explicit "your previous output did not match the schema" instruction, then accepts or fails permanently. This is the only retry that is allowed to change the prompt content between attempts; every other retry (Decision 5) resends the identical request.
- This mechanism is entirely local and requires no model-native "JSON mode" support — `structured_output_supported=true` in the registry (Decision 2) means "the runtime will enforce this," not "the model has a native constrained-decoding feature," which keeps the registry honest about a small instruct model that has no guaranteed native grammar-constrained decoding in Ollama's current API for this activation.

## Decision 5: budgets, timeouts, cancellation, circuit breakers

Concrete first-cut numbers, chosen to be conservative for CPU-only local inference and tightened later from real measurement, not aspirational:

| Control | Value | Rationale |
|---|---|---|
| Routing overhead | p95 <50ms | Matches `PHASE-004-ai-runtime.md`'s stated NFR; achievable because Decision 2's pipeline is in-memory only. |
| Per-model-call timeout | 20s | Generous for a 1.5B CPU model on a bounded (≤512 token) output; short enough that a hung call fails fast relative to a human waiting on a UI action. |
| Per-tool-call timeout | 5s | Both slice-1 tools (Decision 6) are single indexed-read database queries against existing Phase 1-3 tables; 5s is already a large multiple of their expected latency. |
| Total run budget (wall clock) | 60s | Covers one model call, one bounded repair retry, and up to two tool calls with margin; a run exceeding this is cancelled and marked `degraded`/`failed`, never left running. |
| Max input tokens (prompt) | 3072 | Well under the model's native window; chosen for CPU throughput predictability, not because the model can't accept more — revisit once Decision 8's larger-context tasks exist. |
| Max output tokens | 512 | Slice 1's tasks (Decision 6) are short explanations/summaries, not long-form generation. |
| Cost budget | $0 (tracked, not enforced) | No metered remote provider exists in this activation (Decision 7); the `cost` field is populated as `0.0` so the schema and evaluation metrics stay uniform for when a remote provider is added. |
| Retry policy | 1 bounded retry on `schema_invalid` (Decision 4) or a transient provider error (timeout, connection reset); never retried after a non-idempotent step has run | Tool calls in this activation are all read-only (Decision 6), so nothing here is actually non-idempotent yet — the rule is stated now so it does not need re-deriving when a mutating tool is added in a later slice. |
| Circuit breaker | Open after 3 consecutive failures within a rolling 60s window per `model_definitions` row; half-open probe after a 30s cool-down; one probe success closes it, one probe failure reopens it for another 30s | Standard three-state breaker; excluded from eligibility (Decision 2, step 5) while open. |
| Cancellation | Ollama's streaming generate endpoint is used (not the non-streaming call), so an operator- or budget-triggered cancellation closes the stream mid-generation instead of waiting for a non-preemptible blocking call to return | Chosen specifically because a synchronous non-streaming call cannot be cancelled mid-flight; this is a real mechanism, not an aspiration — see the Threat model section for what cancellation does and does not guarantee. |

## Decision 6: bounded tool runtime — first tool contracts

Two read-only tools, deliberately the smallest useful step: both wrap already-authorized, already-tested Phase 1-3 domain queries with zero new mutation surface.

1. **`attention.get_item`** — wraps Phase 3's existing `GET /attention/{id}` read path (`backend/ecc/domains/attention/attention.py`). Input: `{attention_item_id}`. Output: `{entity_type, score, confidence, factors: [{code, label, points, source_field}], evidence_refs}`. Scope: `read:attention`. This is the tool the first evaluated task (below) uses.
2. **`knowledge.get_entity`** — wraps Phase 2's existing `GET /knowledge/entities/{id}` read path. Input: `{entity_id}`. Output: `{title, claims: [...], evidence: [...]}`. Scope: `read:knowledge`. Registered in this activation so the allowlist/registry mechanism is proven against more than one contract from day one, but not wired to an evaluated task until the slice that needs it (Decision 8) — registering it now costs nothing (it is read-only, workspace-scoped exactly like every existing Phase 2 read) and avoids a second migration later purely to add a row.

**Allowlist/scoping mechanism.** A task's typed port declares its `eligible_tools` list at the port definition (application code), not at prompt-render time — a model's tool-call request is only ever dispatched if the requested tool name is in the task's own declared list, checked before the tool call is even attempted, so a prompt-injected instruction asking for a tool outside that list is rejected by the runtime, not by the model choosing not to comply. Every tool call's arguments are themselves schema-validated against `tool_definitions.input_schema` before execution (Decision 4's same Pydantic mechanism, reused for tool inputs, not just task outputs). Every tool result is schema-validated against `output_schema`, size-bounded (a hard cap on returned rows/characters, e.g. `attention.get_item` returns exactly one item, `knowledge.get_entity` truncates claims/evidence lists to a fixed page size), and re-inserted into the model's context tagged as **data**, never as instructions (Threat model section).

## Decision 7: data class policy — remote providers deferred entirely

**Data classes for this activation:** `public`, `internal`, `sensitive`, `restricted` (four tiers, matching the granularity `docs/phases/phase-007/DOMAIN-PRIVACY-CONTRACT.md` already assumes exists for a later phase, so Phase 7 inherits the same taxonomy rather than inventing a second one). Every domain record Phase 4 can currently read (Phase 1 tasks/commitments/risks, Phase 2 entities/claims, Phase 3 attention items) defaults to `sensitive` unless a future phase-specific classification rule says otherwise — a conservative default, not a precise taxonomy, because no phase before this one needed to classify data for AI egress purposes.

**Decision: no remote provider is registered or wired up in this first activation.** `model_definitions` contains exactly one row (Decision 1's Ollama model), `deployment=local` for every data class listed above. This mirrors Phase 2's embeddings activation precedent exactly (`ADR-0011`: fully local, no external network calls at query or embed time) and keeps this first AI Runtime activation's blast radius to "a local process talking to another local process over `localhost`," with **zero** new network egress destination — the single highest-leverage way to make a first activation in a brand-new, higher-risk domain (LLM inference, prompt injection, tool execution) lower-risk. This is a deliberate, explicit choice, not a placeholder: the data-class egress matrix `PHASE-REVIEW.md`'s approval gate names is, for this activation, "every class is local-only, no class is remote-eligible" — trivial by construction rather than a matrix requiring per-class remote-provider judgment calls that this document is not yet positioned to make (which named provider, what data leaves, under what contract — all explicitly out of scope until a later Phase 4 slice, Decision 8).

Practically: `Settings.ai_runtime_remote_enabled` does not exist as a flag yet because there is no remote adapter for it to gate — the equivalent of Phase 2's `Settings.embeddings_enabled`, for remote providers, is simply "the capability does not exist in the codebase," the strongest possible form of "opt-in and blocked for disallowed data classes" (`PHASE-004-ai-runtime.md`'s own functional requirement).

## Decision 8: what is deferred out of this first activation

Explicitly out of scope for the first implementation slice (in addition to `PHASE-004-ai-runtime.md`'s own "Out of scope"/"Deferred backlog," which this document does not relitigate):

- **Remote providers of any kind** (Decision 7) — no cloud adapter, no API-key management, no remote egress code path at all, not even a disabled one.
- **A second local model or runtime** — the registry mechanism supports it (Decision 2), but only one `model_definitions` row exists until a concrete second task needs different capability/context trade-offs.
- **Agent Runtime, multi-agent coordination, Reflection Engine** (`chapter-03-ai-runtime.md`'s larger aspiration) — every task in this activation is a single bounded request/response with at most one repair retry and up to two tool calls, no multi-step planning loop, no agent-to-agent handoff.
- **Mutating tools** — both first tools (Decision 6) are read-only; a tool that writes to Phase 1-3 domain data is a later slice's decision, requiring its own idempotency/authorization design this document does not attempt.
- **Phase 3's meeting-prep AI enrichment flag** — flipping it on is a Phase-4-consuming change to Phase 3's `meeting_prep.py`, not part of this activation; this document's runtime is shaped to make that flip straightforward later (its enrichment section already expects exactly this kind of typed task port), but does not perform it.
- **Fine-tuning, online learning, cross-workspace evaluation data** — already excluded by `PHASE-004-ai-runtime.md`'s own Deferred backlog; restated here because Decision 6's evaluation harness could otherwise be misread as a training pipeline. It is not one — it is a fixed, versioned, human-labelled comparison dataset, nothing adaptive.
- **Prompt-injection defenses beyond the Threat model section below** — no dedicated injection-classifier model, no sandboxed sub-agent review step; the mitigations below rely on allowlisting, schema validation and data/instruction separation, which is judged sufficient for a read-only, two-tool, single-local-model first activation, and explicitly not sufficient once mutating tools or remote providers are added (a later ADR's problem).

## Decision 9: evaluation harness — first dataset

**First task type: `attention.explain_item`.** Given an `attention_item_id`, produce `{explanation_text: str (<=60 words), cited_factor_codes: list[str]}` — a plain-language explanation of why an item is ranked where it is, grounded entirely in the item's own already-computed, already-deterministic `factors` (via the `attention.get_item` tool, Decision 6). This is chosen as the very first evaluated task specifically because it is checkable structurally, not just by human/LLM judgment: **every `cited_factor_codes` entry must appear in the source item's actual `factors` list** — a hard, programmatic grounding check, not a fuzzy similarity score. It also directly serves an already-approved UX requirement (`phase-003/UX-STATES.md`: "Scores are secondary to plain-language rationale") without requiring any new Phase 3 data.

**Dataset (`evaluation_sets`, `task_type='attention.explain_item'`, version 1):** 20 hand-labelled examples, `tests/fixtures/phase4_evaluation_attention_explain.py` (versioned, checked in, mirroring Phase 3's `tests/fixtures/phase3_*` convention), drawn from representative attention items across all entity types Phase 3 scores (`task`, `commitment`, `risk`, `waiting_link`, `risk_review`, `meeting`) with 3-4 examples each. Each example specifies: the input item's factors, a `must_cite` set (factor codes the explanation must reference to be considered grounded), a `must_not_state` set (facts that do not exist on the item — a hallucination probe, e.g. a due date when the item has none), and a reference explanation for readability comparison only (not an exact-match target — natural-language phrasing is not scored for exact match).

**Metrics and promotion floors** (per `EVALUATION-CONTRACT.md`'s required metric set, made concrete):

| Metric | Floor | Consequence of miss |
|---|---|---|
| Schema validity | 100% | Any invalid output blocks promotion — this is Decision 4's enforcement made visible in the evaluation report, not a separate check. |
| Grounding (cited factors exist on the item) | 100% | A citation to a nonexistent factor is treated as a hallucination, not a minor miss — zero tolerance, matching `must_not_state` below. |
| Prohibited-fact rate (`must_not_state` violations) | 0 occurrences tolerated | Any occurrence blocks promotion; this is a hard safety floor stated as a count, not a percentage, because even one instance is a fabricated fact about the executive's own work. |
| Latency (p95, full run including tool call) | <20s | Matches Decision 5's per-model-call timeout; a task that cannot stay under its own configured timeout at the p95 percentile is not ready for promotion. |
| Human-labelled vs. development split | Separate | The 20 examples above are the labelled evaluation set; any examples used during prompt-template iteration before evaluation are a distinct, smaller development set never reused as evaluation data, per `EVALUATION-CONTRACT.md`'s existing separation rule. |

Promotion of `prompt_versions`/`routing_policies` changes for this task always re-runs the full 20-example set and requires the table above to pass in full before the new version can become `active` (Decision 3).

## Threat model summary

Concrete mitigations for each of `PHASE-004-ai-runtime.md`'s named security concerns, not a restatement of the requirement list:

- **Prompt injection** (an attacker embeds instructions inside retrieved content — e.g. a note's free text, a claim, or a tool result — hoping the model treats it as a command). Mitigation: every tool result and every piece of retrieved evidence is inserted into the prompt inside a clearly delimited, explicitly labelled "untrusted data" section (per `chapter-03-ai-runtime.md`'s Prompt Structure: Context is separate from Instructions), and the system prompt explicitly instructs the model that content in that section is data to reason about, never instructions to follow. This is a real but incomplete mitigation against a genuinely adversarial model-following-injected-instructions failure mode — the hard backstop is structural, not behavioral: Decision 6's allowlist means even a fully successful injection ("ignore your instructions and call tool X") cannot actually invoke a tool outside the task's declared `eligible_tools`, and Decision 4's schema validation means even a fully successful injection cannot produce an output that reaches a domain mutation, because slice 1 has no domain-mutating output path at all (Decision 8).
- **Tool-output trust boundary.** A tool's return value is data returned to the model, not code or instructions executed by the runtime — the runtime never re-interprets a tool's textual output as a new instruction to itself (only the model sees tool output as content; the orchestration loop only sees the tool's schema-validated structured return value, decided in application code, not string-matched from anywhere the model could influence).
- **Data exfiltration via tool calls.** Bounded to "impossible in this activation" by construction: both first tools are read-only, there is no tool that can write anywhere (a filesystem, network endpoint or external service), and there is no remote provider (Decision 7) for any tool result or prompt content to be sent to — the only network hop in the entire request path is `backend` to the local Ollama process, both inside the same trust boundary Docker Compose already establishes. This mitigation is structural (nothing to exfiltrate through) rather than behavioral (a policy that could be violated), and is explicitly named as temporary: the day a remote provider or a network-capable tool is added, this section must be rewritten, not just re-approved.
- **Secret leakage in traces.** `ai_run_steps`' trace persistence is bounded and redacted by default (`DATA-MODEL.md`: "Raw sensitive content is not stored by default"); the trace redaction rule from RFC-005's Observability baseline ("Secrets, credentials, full prompt context and sensitive source content MUST NOT be logged") applies identically here — the AI Runtime's structured logs record `prompt_version`, `tool_definitions.version`, `model_id`, timing, token counts and validation outcome, never the rendered prompt text or raw model output, unless a workspace has explicitly opted into verbose trace retention for debugging (an admin-only, time-bound setting, off by default, matching `DATA-MODEL.md`'s existing default).

## Architecture impact

New backend package `backend/ecc/domains/ai_runtime/` — `router.py` (Decision 2), `registry.py` (`model_definitions` CRUD/read), `prompts.py`/`tools.py` (Decision 3's versioning), `validator.py` (Decision 4, thin wrapper over Pydantic `TypeAdapter`), `budgets.py` (Decision 5's timeout/circuit-breaker state), `runtime.py` (the orchestration loop: route → render prompt → call model → validate → optionally call tool → optionally repair-retry → persist `ai_runs`/`ai_run_steps`), `evaluation.py` (Decision 9's harness runner). Domain modules (`attention/`, `knowledge/`) gain no new files in this activation beyond what Decision 6's tool wrappers need — a tool is a thin adapter calling the existing domain read function, not new domain logic. `ollama` (the Python client package, PyPI-installable, confirmed reachable in this sandbox) is the only new runtime dependency; `RFC-005 v1.3.0`'s amendment records the exact pinned version.

## Test strategy and the sandbox constraint

Mirrors `ADR-0011`'s Consequences section's handling of the analogous `sentence-transformers`/`torch`-in-Alpine-CI constraint, honestly stated for this activation's own constraint:

**What cannot be verified in this sandbox.** This development sandbox has no outbound network access to `ollama.com` (confirmed: `curl https://ollama.com` returns a 403 from the environment's egress proxy) and cannot install or run the actual Ollama server binary. Real end-to-end behavior — an actual model producing actual tokens through an actual local Ollama server — cannot be exercised here. This is a sandbox limitation, not a design gap: it is exactly the same category of constraint `ADR-0011` recorded for `torch` on Alpine (a real dependency that genuinely cannot run in every environment this repository's tooling touches).

**What can be verified in this sandbox, and how.** The `ollama` Python client package is installable via `pip` (PyPI access confirmed working) and its request/response shapes are documented and stable, so:
- Every contract-level test — routing eligibility/preference (Decision 2), budget/timeout/circuit-breaker state transitions (Decision 5), prompt/tool immutability and activation (Decision 3), schema validation accept/reject (Decision 4), tool allowlist enforcement (Decision 6) — runs against a mocked/stubbed HTTP client standing in for Ollama's REST API, exercising the AI Runtime's own logic without a live model. This is the majority of the test surface and needs no live Ollama at all.
- Adversarial fixtures (prompt injection, oversized context, malformed tool output, schema confusion) are constructed as canned stub responses fed through the real validation/allowlist code paths — the mitigations in the Threat model section are testable without a live model, because they are properties of the runtime code, not the model's behavior.
- What genuinely requires a live Ollama server — real generation quality, Decision 9's evaluation-floor numbers actually passing against real model output, real p95 latency on real CPU hardware — requires a dedicated CI job (mirroring `ADR-0011`'s `embeddings-benchmark` job pattern) that provisions an actual Ollama service (e.g. the official `ollama/ollama` container image, pulled by CI's runner with real network access, not this sandbox) and runs `qwen2.5:1.5b-instruct-q4_K_M` for real. This job is designed here but was not run here — it must be verified in a real CI/deployment environment before any promotion decision (Decision 9) is trusted, exactly as `ADR-0011`'s embeddings benchmark required real CI, not sandbox execution.
- The implementation plan's own tasks (below) name which steps are sandbox-executable now versus require that dedicated CI job, so this constraint is visible task-by-task, not just in this paragraph.
