# Phase 2 Knowledge Platform Design

**Status of this document:** planning artifact only. It does not change `docs/phases/PHASE-002-knowledge-platform.md`'s `status: Draft`, does not start implementation, and does not close any Phase 1 exit gate. Per `docs/ROADMAP.md`'s approval gates, Phase 2 implementation may not begin until: Phase 1 exits (seven-day daily-use validation and human change-review sign-off, both still open as of this writing), Phase 2's supporting contracts move from Draft to Approved for Implementation, and the architectural decision in "Open decision 1" below is resolved by the repository owner. This document exists to make that approval decision informed, and to leave an implementation-ready plan queued for the moment it's granted.

## Outcome

Turn the read-only, effectively-unused PKOS foundation shipped in Phase 0 into a working knowledge platform: canonical people/organizations/projects/topics/decisions/documents, typed relationships, atomic attributable claims, deterministic entity resolution with human review, reversible merge/split, timeline projection, and lexical retrieval (hybrid retrieval optional and gated). Scope is exactly `docs/phases/PHASE-002-knowledge-platform.md`; this document does not expand it.

## Why this isn't a green field

Phase 1 shipped `pkos_nodes`, `pkos_edges`, and `pkos_evidence` (`backend/migrations/versions/0001_foundation.py`), and ADR-0003 (Accepted) already declared PKOS "the canonical knowledge subsystem." But nothing in Phase 1 ever wrote to `pkos_nodes` or `pkos_edges` — the only production code that touches PKOS tables is `backend/ecc/domains/knowledge/evidence.py` (a read-only `GET /api/v1/evidence` join against `pkos_evidence`/`pkos_nodes`) and a `pkos_nodes`/`pkos_evidence` existence check in `backend/ecc/domains/communication/commitments.py`. The only rows that have ever existed in these tables come from `scripts/seed_phase1_acceptance.py`, a synthetic fixture used solely to prove backup/restore checksums. Phase 2 is therefore not "add a new subsystem" — it's "make the subsystem Phase 0 scaffolded and Phase 1 never used actually do something," while an entity/relationship-shaped record already exists that other code (evidence resolution) already depends on.

## Open decision 1 (blocking): reconcile `phase-002/DATA-MODEL.md` with the already-Accepted PKOS schema

This is the most important finding in this document and needs the repository owner's decision before contracts can move to Approved for Implementation.

**The conflict:** `docs/domain/PKOS-SCHEMA.md` (Accepted, v1.1.1) is the normative logical-to-physical mapping for PKOS. It already anticipates Phase 2 explicitly — its mapping table lists "aliases and merge records" and "embeddings and chunks" as "deferred until identity-resolution phase" / "explicitly deferred," i.e. deferred to *this* phase, not invented fresh by it. It states in its own architectural rules: *"`pkos_nodes` and `pkos_edges` are rebuildable projections and relationships, not a second aggregate store."* Meanwhile `docs/phases/phase-002/DATA-MODEL.md` (Draft, v0.1.0) independently proposes ten brand-new tables — `knowledge_entities`, `entity_aliases`, `knowledge_claims`, `relationships`, `source_refs`, `resolution_candidates`, `entity_operations`, `timeline_entries`, `retrieval_documents`, `embedding_projections` — that do not reuse `pkos_nodes`/`pkos_edges`/`pkos_evidence` at all. Built as written, Phase 2 would stand up a second, disconnected entity/relationship store while `pkos_nodes`/`pkos_edges` sit permanently empty and `pkos_evidence` keeps being queried by the unrelated, still-live `evidence.py` code path. That directly contradicts PKOS-SCHEMA's "not a second aggregate store" rule and ADR-0003's "adopt PKOS as the canonical knowledge subsystem" decision — both Accepted, neither superseded.

There's also a second, smaller gap: PKOS-SCHEMA.md's mapping table describes column additions (`entity_id`, `status`, `confidence`, `version`, `valid_from`/`valid_to`, `evidence_id` FK on edges) as a "Phase 1 migration" that should already have happened. It never did — `git log`/`grep` across every Phase 1 migration (`0001` through `0009`) shows no ALTER TABLE ever touched `pkos_nodes` or `pkos_edges` beyond their Phase 0 creation. This isn't a Phase 2 regression; it's a pre-existing gap between an Accepted doc and what shipped, surfaced now because Phase 2 is the first phase that would actually need those columns.

**Recommendation: reconcile, don't fork.** Extend the existing `pkos_nodes`/`pkos_edges`/`pkos_evidence` tables to carry the Phase 1-deferred columns PKOS-SCHEMA.md already specifies, and layer Phase 2's genuinely new concepts (claims, resolution, merge/split lineage, timeline, retrieval documents, embeddings) as new tables that reference them — instead of inventing parallel `knowledge_entities`/`relationships` tables. This is the option that requires no new ADR (it fulfills ADR-0003/PKOS-SCHEMA as written, it doesn't reverse them) and keeps `evidence.py`'s existing query path meaningful instead of orphaned. The alternative — formally retiring PKOS in favor of DATA-MODEL.md's independent naming — is a legitimate choice too, but it reverses an Accepted ADR and requires a new one (`docs/adr/ADR-0003-knowledge-platform-pkos.md`'s own status would need to change), which is a materially bigger and slower approval path for no evident benefit: nothing in DATA-MODEL.md's ten tables is impossible to express as an extension of the existing three.

Concretely, the reconciled mapping this document recommends `phase-002/DATA-MODEL.md` adopt before approval:

| DATA-MODEL.md proposal | Reconciled with PKOS-SCHEMA.md |
|---|---|
| `knowledge_entities` | Extend `pkos_nodes` in place: add `entity_id UUID` (nullable, links to an owning domain aggregate when the entity mirrors one — e.g. a Person created from a Commitment counterparty), `status` (`active\|archived\|redirected`), `confidence NUMERIC(3,2)`, `version BIGINT`. `node_type`/`canonical_name`/`attributes` already exist and match `kind`/`canonical_name`/free-form fields. |
| `entity_aliases` | New table (PKOS-SCHEMA explicitly deferred this to now) — `entity_id` FK to `pkos_nodes`. |
| `knowledge_claims` | New table — `subject_id` FK to `pkos_nodes`, distinct from relationships because a claim's object is a scalar/JSON value, not another entity. |
| `relationships` | Extend `pkos_edges` in place: add `confidence NUMERIC(3,2)`, `evidence_id` (nullable composite FK to `pkos_evidence`), `valid_from`/`valid_to`, `status` (`active\|disputed\|invalidated`) — exactly the columns PKOS-SCHEMA.md's mapping table already names. `source_node_id`/`target_node_id`/`edge_type`/`attributes` already exist and match `from_entity_id`/`to_entity_id`/`relationship_type`/free-form fields. |
| `source_refs` | `pkos_evidence` already is this table; PKOS-SCHEMA.md calls it "active authoritative evidence metadata." Add the `evidence_state` enum (`available\|missing\|permission_denied\|deleted`) and `observed_at` PKOS-SCHEMA's evidence contract already specifies but the Phase 0 migration never added. |
| `resolution_candidates`, `entity_operations`, `timeline_entries`, `retrieval_documents`, `embedding_projections` | Genuinely new — PKOS-SCHEMA has no equivalent for any of these. Add as proposed, with `entity_id`/`left_entity_id`/`right_entity_id`/etc. as composite FKs into `pkos_nodes`. |

This table is the concrete edit `phase-002/DATA-MODEL.md` needs before it can move to Approved for Implementation. It is *not* applied to that file by this document — changing an Approved-track contract is the repository owner's call, not something to slip in silently under a "design doc" label.

**What this does not resolve:** whether `evidence.py`'s current ad hoc SQL join should be rewritten against the extended schema (recommended, since it becomes trivial once `evidence_state` exists) — left to the implementation slice that touches evidence, not a blocking decision.

## Delivery strategy

Same phase-then-slice discipline as Phase 1's plan: one vertical, contract-preserving slice at a time, red-green-refactor per behavior, full regression before each commit. Slices below map 1:1 to `docs/phases/phase-002/IMPLEMENTATION-STATUS.md`'s eight planned slices; this document doesn't renumber or change that table, only makes each row concrete enough to execute.

Ordering rationale: entities/aliases/claims (Slice 1) must exist before anything can reference them. Relationships (Slice 2) before timeline (Slice 3), since timeline entries are partly derived from relationship/claim history. Resolution (Slice 4) before merge/split (Slice 5), since merge always starts from a confirmed resolution candidate. Lexical retrieval (Slice 6) before hybrid (Slice 7), matching the contract's "lexical is the mandatory release floor" rule — Slice 7 is the only slice gated on an external approval (see below) and can slip a full release without blocking the rest. Frontend (Slice 8) is threaded through every slice's acceptance criteria rather than deferred entirely to the end, matching Phase 1's approach of not leaving a giant untested UI slice for last — see the implementation plan for per-slice UI scope.

## Architecture impact

New backend package: `backend/ecc/domains/knowledge/entities.py` (Slice 1 queries/mutations), `entities_aliases.py`, `claims.py`, `relationships.py` (Slice 2), `timeline.py` (Slice 3), `resolution.py` (Slice 4), `entity_operations.py` (merge/split, Slice 5), `retrieval.py` (Slice 6-7). `evidence.py` (existing) gains the `evidence_state` column read and is extended, not replaced. Person/Organization creation endpoints live under `backend/ecc/domains/identity/` per `docs/domain/DOMAIN-MODEL.md`'s ownership map (Identity owns Person/Organization; Knowledge Platform owns Note/KnowledgeItem/Document/Decision/Relationship/Evidence) — this is a new top-level domain package, since Identity currently only has `Workspace`/`User` handled inline in `ecc/auth.py` with no dedicated module. No new runtime dependency: PostgreSQL full-text/trigram search only for Slice 6, matching RFC-005/PKOS-SCHEMA's Phase 1 search boundary carried forward. Rebuildable projections (`timeline_entries`, `retrieval_documents`, `embedding_projections`) get a `scripts/rebuild_knowledge_projections.py` entry point mirroring the existing rebuild-from-source pattern implied by PKOS-SCHEMA's "projections are rebuildable" rule — no new scheduler/worker technology, invoked the same way `scripts/seed_phase1_acceptance.py` and `scripts/phase1_evidence.py` are invoked today (CLI, then wired into CI/runbooks).

## Entity resolution approach

Implements `docs/phases/phase-002/ENTITY-RESOLUTION-CONTRACT.md`'s five-level match hierarchy as a pure, unit-testable scoring function (`resolution.py:score_candidate`) separated from the transactional confirm/reject mutation, mirroring how `backend/ecc/domains/governance/attention.py` already separates deterministic ranking logic from mutation. Concretely:

1. Levels 1-4 (confirmed mapping, trusted external ID, verified normalized ID, alias+kind) are deterministic lookups against `entity_aliases` — no scoring, no candidate row, an alias just attaches.
2. Level 5 (weighted fuzzy) is the only level that ever creates a `resolution_candidates` row. Factors: normalized-name trigram similarity (`pg_trgm`, already implicitly available — Phase 1's `SEARCH-CONTRACT.md` already uses "approved trigram"), alias overlap count, shared-relationship-neighborhood count (count of common `pkos_edges` neighbors), temporal compatibility (no conflicting `valid_from`/`valid_to` on claims that would make the same person "employed at two places full-time" simultaneously, as a cheap plausibility filter, not a hard rule).
3. Every candidate row stores `factors_json` with the raw per-factor contributions and a `resolver_version` — required by the contract, and what makes threshold changes reviewable via before/after benchmark rather than by reading code diffs.
4. Thresholds (what score creates a candidate vs. auto-attaches vs. is dropped) are typed configuration (a small dataclass, not hardcoded floats scattered through the function), because the contract requires "benchmark evidence before change" — a config value is the unit that gets benchmarked and reviewed, not a code change.

**Explicitly not built:** any ML/embedding-based resolution signal in the first pass — Slice 4 ships resolution using only lexical/graph/temporal factors, all available without the RFC-005 gate below. If evaluation against the labelled dataset (Slice 4's exit gate) shows lexical-only resolution has an unacceptable false-merge or unresolved rate, that's a signal to revisit thresholds or add factors, not a signal to reach for embeddings without going through Open decision 2.

## Merge/split and reversibility

`entity_operations` records `merge_id, operation_type, inputs_json (source entity IDs + their pre-merge state snapshot), outputs_json (target entity ID + redirect map), actor_id, reason`. Merge is one transaction: redirect `pkos_nodes.status='redirected'` on sources, rehome `entity_aliases`/active `pkos_edges` to the target (deduplicating edges that become identical post-redirect — e.g. two "WORKS_ON" edges to the same project), write the operation record, audit event, and outbox event. Reversal replays the snapshot captured in `inputs_json` and requires validating no dependent operation since the merge would make un-merging unsafe (e.g. a new claim was recorded against the merged target that has no clear source-entity attribution) — this validation is exactly `ENTITY-RESOLUTION-CONTRACT.md`'s "reversal restores prior identities unless a later dependent operation requires manual split" rule, implemented as a pre-reversal check that returns the `unsafe_reversal` error code from `API-SCHEMAS.md` rather than silently partial-reversing.

No dedicated graph database, and no ADR is sought for one in this document — `docs/phases/PHASE-002-knowledge-platform.md`'s own "Out of scope" already excludes it, and nothing in merge/split or the resolution neighborhood-scoring above needs more than the SQL adjacency queries `pkos_edges` already supports at the 10k-entity acceptance scale the non-functional requirements target.

## Retrieval approach

Slice 6 (mandatory) implements exactly `RETRIEVAL-CONTRACT.md`'s pipeline steps 1-3 and 6-7 (normalize, exact-identifier/alias match, PostgreSQL full-text + trigram, filter, explain) against `retrieval_documents` — a denormalized, rebuildable projection populated from `pkos_nodes`/`entity_aliases`/`knowledge_claims`/relationship summaries at write time via the same deferred-until-commit pattern Phase 1 already uses for metrics (`queue_lifecycle_event` et al. in `backend/ecc/observability.py`), so a rolled-back mutation never leaves a stale projection. Fusion/ranking for lexical-only mode is just PostgreSQL's native `ts_rank` plus the bounded authority/recency/confirmed-status adjustments the contract allows — no custom ranking algorithm to design or benchmark beyond weight tuning.

## Open decision 2 (not blocking Slices 1-6): embeddings/hybrid retrieval requires an RFC-005 amendment before Slice 7 starts

`docs/RFC-005.md`'s technology registry lists `pgvector` under "Approved later, not Phase 0 dependencies" with an explicit activation requirement: "Retrieval benchmark and ADR." `docs/phases/PHASE-002-knowledge-platform.md` itself says the same thing ("Optional embeddings are derived, local-first and gated by RFC-005/ADR approval"). This document does not attempt that benchmark or draft that ADR — Slice 7 is the only slice in this plan that cannot start until: (a) Slices 1-6 are live long enough to produce a real retrieval-quality baseline to benchmark against, and (b) the repository owner decides embeddings are worth pursuing at all given `docs/architecture/chapter-04-knowledge-platform.md`'s explicit "if embeddings fail, graph traversal continues" fallback-first philosophy is already fully satisfied by Slice 6 alone. Recommendation: treat Slice 7 as released independently, after Slices 1-6 ship and exit, not bundled into the same release.

## Frontend approach

New `frontend/src/features/knowledge/` mirroring the existing `features/{tasks,notes,risks}` structure: `EntityExplorer.tsx` (list/search, Slice 6 dependent), `EntityDetail.tsx` (claims/relationships/timeline/provenance, Slices 1-3), `ResolutionInbox.tsx` (Slice 4), `MergeReview.tsx` (Slice 5). Per `UX-STATES.md`'s explicit rule, relationship/timeline views ship an accessible list/table as the primary representation — no graph-visualization library evaluation or dependency needed for release, which also keeps this slice inside RFC-005's existing frontend baseline (no new package). `frontend/src/api/types.ts` gains the Phase 2 entity/claim/relationship/resolution types; `frontend/src/api/client.ts`'s existing `apiRequest`/CSRF/idempotency/conflict machinery is reused as-is — Phase 2 introduces no new transport concern Phase 1 didn't already solve.

## Events and observability

New catalog entries for `docs/domain/EVENT-CATALOG.md` (proposed, not yet added to that Accepted doc — that edit belongs to Slice 1's PR, reviewed alongside the code that emits them, same as how Phase 1's events were added incrementally): `knowledge_entity.created.v1`, `knowledge_entity.claim_recorded.v1`, `knowledge_entity.archived.v1`, `knowledge_entity.restored.v1`, `relationship.created.v1`, `relationship.invalidated.v1`, `resolution_candidate.created.v1`, `resolution_candidate.confirmed.v1`, `resolution_candidate.rejected.v1`, `entity_operation.merged.v1`, `entity_operation.reversed.v1`. Metrics follow Phase 1's deferred-until-commit pattern: projection lag, unresolved resolution count, merge/reversal outcomes, retrieval latency by mode, degraded-fallback count — exactly `PHASE-002-knowledge-platform.md`'s "Observability" section, each counter queued via `session.info` and flushed only in `after_commit`, matching `backend/ecc/observability.py`'s existing `queue_brief_generated`/`queue_recommendation_transition` helpers rather than inventing a new metrics pattern.

## Test strategy

Follows `phase-002/TEST-PLAN.md` directly; the one addition this document makes concrete is *where* the entity-resolution labelled dataset and retrieval benchmark live: `tests/fixtures/phase2_resolution_dataset.py` and `tests/fixtures/phase2_retrieval_benchmark.py`, versioned and checked in (not generated ad hoc per test run), mirroring how `scripts/seed_phase1_acceptance.py` is a checked-in deterministic fixture rather than randomly generated. Property/adversarial tests (alias collisions, Unicode normalization, circular relationships, concurrent merges) run against real PostgreSQL, matching every existing `test_*_postgres.py` convention in `tests/` — no new test infrastructure.

## Non-goals (restating PHASE-002's "Out of scope" for implementation-time visibility)

External connectors, autonomous merges, a dedicated graph database, cross-workspace learning, background agents, predictive recommendations, multi-user collaboration, mandatory cloud vector services, generative-AI-authored claims. Also explicitly not in this document's scope: resolving Open decision 1 or 2 (both require the repository owner), or touching any Phase 1 code path outside what's named above.
