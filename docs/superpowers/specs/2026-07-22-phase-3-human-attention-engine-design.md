# Phase 3 Human Attention Engine Design

**Status of this document:** planning artifact only. It does not change `docs/phases/PHASE-003-human-attention-engine.md`'s `status: Draft`, does not start implementation, and does not close any Phase 1/2 exit gate. Per `docs/ROADMAP.md`'s approval gates, Phase 3 implementation may not begin until: its dependencies' exit criteria are evidenced (or the repository owner grants the same kind of parallel-start exception Phase 2 received), `docs/phases/phase-003/*.md` contracts move from Draft to Approved for Implementation, and Open decision 1 below is resolved by the repository owner. This document exists to make that approval decision informed, and to leave an implementation-ready plan queued for the moment it's granted.

## Outcome

Turn trusted Phase 1 work (tasks, commitments, risks) and Phase 2 knowledge (entities, relationships, claims) into an explainable, user-controlled attention system: unified ranking with visible factors, waiting direction, risk review cadence, capacity-aware daily/weekly planning with conflict-transparent proposals, and evidence-backed meeting preparation. Scope is exactly `docs/phases/PHASE-003-human-attention-engine.md`; this document does not expand it.

## Why this isn't a green field

Phase 1 already shipped a working, in-production attention system â€” `backend/ecc/domains/governance/attention.py` (578 lines) and its `attention_items` table (`backend/migrations/versions/0006_phase1_risks_priority.py`). It ranks tasks/commitments/risks with per-type point-based factors (`_score_task`/`_score_commitment`/`_score_risk`), exposes `GET /api/v1/attention`, `POST /api/v1/attention/regenerate`, and `dismiss`/`defer`/`restore` mutations, and already writes `attention_item.dismiss`/`defer`/`restore` audit events. `docs/domain/DOMAIN-MODEL.md:108` ("Derived knowledge, attention projections and summaries are rebuildable and never replace evidence") and `docs/domain/EVENT-CATALOG.md`'s existing `attention_item.created.v1`/`attention_item.updated.v1` rows both already anticipate this being extended by a later phase. Phase 3 is therefore not "add attention ranking" â€” it's "make the attention system Phase 1 shipped cover waiting/risk-review/meetings/planning too," while a same-named table and module with different, incompatible shape already exists.

## Open decision 1 (blocking): reconcile `phase-003/DATA-MODEL.md` with the already-shipped `attention_items` table

This is the most important finding in this document and needs the repository owner's decision before contracts can move to Approved for Implementation â€” the same class of problem Phase 2's design doc found with PKOS, resolved the same way here: extend, don't fork.

**The conflict:** `docs/phases/phase-003/DATA-MODEL.md` (Draft, v0.1.0) independently proposes a new `attention_items` table (`source_type, source_id, state, score, confidence, policy_version, factors_json`) plus a separate `attention_overrides` table (`attention_item_id, action, reason, effective_until, actor_id`) for pin/dismiss/defer. Both names and shapes diverge from what already exists and is live:

| Phase 3 DATA-MODEL.md proposes | Already shipped (`attention_items`, migration `0006`) |
|---|---|
| `attention_items.source_type` / `source_id` | `entity_type` / `entity_id` (same concept, different names) |
| `attention_items.state` | No `state` column â€” status is derived from `dismissed_at`/`dismissed_entity_version`/`deferred_until`/`expires_at` being null or not, computed at query time in `list_attention`'s `WHERE` clause |
| `attention_items.policy_version` | Does not exist â€” every weight/cap is an inline literal in `_score_task`/`_score_commitment`/`_score_risk`, not a stored or versioned config |
| `attention_items.factors_json` | `factors` (`JSONB`, same shape: list of `{code, label, points, source_field}`) |
| New `attention_overrides` table (action + reason + actor + effective_until) | No separate table â€” `dismissed_at`, `dismissed_entity_version`, `deferred_until` are columns directly on `attention_items`; `pinned` is not stored there at all, it's read live from the source entity's own `pinned` column via `LEFT JOIN`; every dismiss/defer/restore call already writes an `audit_events` row with actor/timestamp, but there is no free-text `reason` field anywhere |

Built as written, Phase 3 would either collide with the existing `attention_items` table name at migration time or stand up a second, disconnected ranking system while the live one keeps running unchanged â€” both wrong, for the same reason Phase 2's independent `knowledge_entities` proposal was wrong against PKOS.

**Recommendation: reconcile, don't fork.**

1. **Extend `attention_items` in place**, not rename or replace it. `entity_type` is a plain `String(32)` with no `CHECK` constraint restricting its values (verified: migration `0006` only constrains `score` and `confidence`) â€” broadening it to accept `waiting_link`, `risk_review`, `meeting` alongside the existing `task`/`commitment`/`risk` is a pure application-layer change (widen the `EntityType` `Literal` in `attention.py`), not a migration. Add two columns: `policy_version SMALLINT NOT NULL DEFAULT 1` (so every existing row is implicitly policy v1 the day this ships â€” no backfill ambiguity) and `override_reason TEXT NULL` (the one genuinely missing piece from the `attention_overrides` proposal: free-text reason capture on dismiss/defer, nullable so it stays optional).
2. **Do not add a separate `attention_overrides` table.** `audit_events` already captures actor + timestamp + action for every dismiss/defer/restore (verified in `_mutate_attention`'s existing `INSERT INTO audit_events` call) â€” that already is the append-only override history DATA-MODEL.md's table would duplicate. `effective_until` is redundant with the existing `deferred_until` column. Keep pin as derived-from-source-entity, matching Phase 1's existing design choice: it avoids a second source of truth for pin state, and nothing in `ATTENTION-MODEL.md`'s override semantics requires pin to carry its own reason or expiry.
   - **This directly conflicts with `phase-003/API-SCHEMAS.md:16`**, which lists `POST /attention/{id}/pin` as part of Phase 3's proposed surface â€” a conflict this document did not originally surface and is naming explicitly now. Recommendation: drop `/pin` from `API-SCHEMAS.md`'s endpoint list as part of Task 0's contract edit; pinning continues to happen exclusively through the source entity's own `PATCH` (task/commitment/risk today, plus whatever Phase 2 entity types Phase 3 scores). If the repository owner instead wants a dedicated pin action independent of the source entity (e.g. to pin a waiting link or meeting, which have no existing `pinned` column of their own to patch), that reopens Open decision 1's schema shape â€” `waiting_links`/`meeting_packs`/`risk_reviews` would each need their own `pinned` column, or `attention_items` would need to become the authoritative pin store instead of a read-through. Flagged for Task 0, not resolved here.
3. **Promote the existing inline scoring into a versioned policy**, since that's the one real gap: `_score_task`/`_score_commitment`/`_score_risk`'s point values become a typed, named policy-v1 config object (weights/caps as data, not literals scattered through three functions) with the *exact same point values* Phase 1 ships today, so no existing ranking silently changes the day this lands. New factors this phase adds â€” dependency/blocked-by, waiting direction, meeting proximity, user-set importance, bounded recency, bounded deferral penalty (`ATTENTION-MODEL.md`'s formula) â€” are then addable as new terms to the same policy-v1 config (additive, regression-tested against Phase 1's existing scenarios) rather than a policy v2 bump, unless the repository owner decides a version bump is cleaner for auditability. That specific call â€” additive v1 vs. v2 bump â€” is folded into "Attention policy weights/caps," the approval gate `docs/phases/PHASE-REVIEW.md:128` already names for Phase 3; this document recommends additive v1 and leaves the final call to Task 0.

Concretely, the reconciled migration this document recommends `phase-003/DATA-MODEL.md` adopt before approval:

| DATA-MODEL.md proposal | Reconciled with the shipped `attention_items` |
|---|---|
| `attention_items` (new) | Extend existing: add `policy_version SMALLINT NOT NULL DEFAULT 1`, `override_reason TEXT NULL`. Keep `entity_type`/`entity_id`/`factors` names as-is (functionally identical to `source_type`/`source_id`/`factors_json`) â€” renaming live columns is churn with no behavior benefit. Widen `EntityType` in `attention.py` to include `waiting_link`, `risk_review`, `meeting`. |
| `attention_overrides` (new table) | Dropped. `audit_events` (already written) plus the new `override_reason` column on `attention_items` covers the same need without a second table. |
| `waiting_links`, `risk_reviews`, `capacity_profiles`, `planning_constraints`, `plans`, `plan_blocks`, `meeting_packs`, `attention_feedback` | Genuinely new â€” no Phase 1/2 equivalent exists for any of these. Add as proposed, workspace-scoped, with composite FKs into `attention_items`/`pkos_nodes`/`calendar_events` as appropriate (see Architecture impact below). |

This table is the concrete edit `phase-003/DATA-MODEL.md` needs before it can move to Approved for Implementation. It is *not* applied to that file by this document â€” changing an Approved-track contract is the repository owner's call, matching Phase 2's precedent.

**What this does not resolve:** whether `_score_task`/`_score_commitment`/`_score_risk`'s refactor into policy-v1 config should also change `attention.py`'s current three-separate-functions shape into one shared scorer â€” left to the implementation slice that touches it (Task 1), not a blocking decision.

## Delivery strategy

Same phase-then-slice discipline as Phase 1 and Phase 2's plans: one vertical, contract-preserving slice at a time, red-green-refactor per behavior, full regression before each commit. Slices below map 1:1 to `docs/phases/phase-003/IMPLEMENTATION-STATUS.md`'s eight planned slices; this document doesn't renumber that table, only makes each row concrete enough to execute.

Ordering rationale: attention policy (Slice 1) must exist and be versioned before anything else references a `policy_version` or extends `attention_items`'s `entity_type`. Waiting (Slice 2) and risk review (Slice 3) are independent of each other but both extend the Slice 1 attention surface, so both come next. Capacity/constraints (Slice 4) must exist before planning (Slice 5) can allocate against them. Plan proposal (Slice 5) before edit/accept/replan (Slice 6), since editing requires a proposal to edit. Meeting preparation (Slice 7) is independent of planning and could in principle run in parallel with Slices 4-6, but is sequenced last among the backend slices because it's the only one needing the new `meeting_participants` table (Open decision 2) and because `MEETING-PREP-CONTRACT.md`'s citation/staleness discipline benefits from the same evidence-freshness patterns Slice 3's risk review and Slice 2's waiting links will have already exercised. Frontend (Slice 8) is threaded through every slice's acceptance criteria rather than deferred entirely to the end, matching Phase 1 and Phase 2's approach â€” see the implementation plan for per-slice UI scope.

## Architecture impact

New backend package: `backend/ecc/domains/attention/` â€” this document recommends *moving* `governance/attention.py` here (Phase 3 owns and substantially extends it; keeping it under `governance/` next to `risks.py` while `waiting.py`/`planning.py`/`meeting_prep.py` sit in a new `attention/` package would split one coherent surface across two packages for no reason) â€” plus new modules `waiting.py` (Slice 2), `risk_reviews.py` (Slice 3), `capacity.py`, `planning_constraints.py` (Slice 4), `planning.py` (Slice 5-6), `meeting_prep.py` (Slice 7). `docs/domain/DOMAIN-MODEL.md`'s ownership map does not currently list an "Attention" domain package explicitly â€” this move and the new package name are a contract update Task 0 should fold in alongside the `attention_items` reconciliation, not a silent restructure. Risks (`governance/risks.py`) stays where it is; Slice 3 only adds a `risk_reviews` history table and a `POST /risks/{id}/review` endpoint reading/writing it, no changes to `risks.py`'s existing CRUD shape. No new runtime dependency for Slices 1-7: planning and scheduling arithmetic (capacity allocation, conflict detection, timezone/DST handling) is deterministic Python over `zoneinfo` (already used by `attention.py`'s `_workspace_day`), matching RFC-005's existing baseline â€” no new date/scheduling library. Rebuildable projections (`attention_items` scores, `plans`' *proposed* state before acceptance) get rebuild entry points mirroring `scripts/rebuild_knowledge_projections.py`'s established pattern; a new `scripts/rebuild_attention_projections.py` is proposed for `attention_items`/re-scoring, since the existing `attention.py:regenerate_attention` endpoint already *is* that rebuild logic â€” the script becomes a thin CLI wrapper over the same function, not new logic.

## Attention policy approach

Implements `docs/phases/phase-003/ATTENTION-MODEL.md`'s deterministic formula as a versioned, typed policy config (`attention/policy.py`, mirroring how Phase 2's `resolution.py:score_candidate` separated pure scoring from mutation): one dataclass per policy version holding every weight/cap named in the formula (`urgency`, `commitment`, `dependency`, `risk`, `meeting`, `importance`, `bounded_recency`, `bounded_deferral_penalty`), with policy v1 seeded from Phase 1's exact current point values (Open decision 1). `_score_task`/`_score_commitment`/`_score_risk` become thin adapters that pull weights from the active policy instead of inline literals; two new scorers (`_score_waiting`, `_score_meeting`) follow the same shape for Slice 2/7's new `entity_type`s. Confidence remains separate from score, declining with missing/conflicting/stale evidence exactly as `ATTENTION-MODEL.md` requires â€” Phase 1's existing per-type confidence logic (e.g. commitment confidence capped at 0.8 when a due date exists) carries forward unchanged into policy v1.

## Waiting direction approach

`waiting_links` (new) records `subject_type/id` (task, commitment, or knowledge entity â€” Phase 2's `pkos_nodes`), `counterparty_entity_id` (FK into Phase 2's extended `pkos_nodes`, always a person/organization), `direction` (`waiting_on_me|waiting_on_them|blocked_by|delegated`), `status`, `since_at`, `expected_at`. Direction changes append a new row rather than mutate in place (`ATTENTION-MODEL.md`: "Direction changes create history; they do not overwrite the original obligation") â€” the *current* direction for a subject is the latest non-superseded row, mirroring Phase 2's claim-supersede pattern (`claims.py`) rather than inventing a new history mechanism. Waiting items surface in the unified attention queue as `entity_type='waiting_link'` rows in the reconciled `attention_items` table (Open decision 1), scored by the new `dependency`/`waiting` policy factors.

## Planning approach

Implements `docs/phases/phase-003/PLANNING-CONTRACT.md`'s seven-step deterministic order as a pure function (`planning.py:propose_plan`) over `capacity_profiles`, `planning_constraints`, hard calendar blocks (`calendar_events`), and the current attention ranking â€” separated from the transactional accept/supersede mutation, same pure/impure split as Phase 2's `resolution.py`. Effort estimates: `PLANNING-CONTRACT.md` requires "the planner does not invent effort... items without estimates use a visible default bucket and lower confidence" â€” Phase 1/2 have no effort-estimate field on tasks/commitments/entities today, so Slice 5 must either add one (a small additive column on `tasks`/`commitments`) or rely entirely on the default-bucket path for this phase's first release. This document recommends the latter (default-bucket only, no new effort field) to avoid a Phase 1/2 schema change inside a Phase 3 slice; revisit as a Phase 3 follow-up if the two-week dogfood (Slice 8) shows default-bucket planning is too coarse to be useful. Replanning produces an explicit added/removed/moved/unchanged/newly-conflicted diff (`plan_blocks` versioned by `plans.version`, comparing the new proposal's blocks against the prior accepted version) â€” no new diff library, a plain per-block-id comparison over two versions' `plan_blocks` rows.

## Meeting preparation approach

Implements `docs/phases/phase-003/MEETING-PREP-CONTRACT.md`'s required deterministic sections by composing already-existing Phase 1/2 queries â€” recent timeline (Phase 2 `timeline.py`), open commitments by direction (Phase 1 `commitments.py` plus Slice 2's `waiting_links`), prior decisions and relationships (Phase 2 `relationships.py`/`claims.py`), active risks (`risks.py`), evidence freshness (`evidence.py`) â€” into one `meeting_prep.py:build_pack` function, snapshotting the source IDs/versions it read (`meeting_packs.source_versions`) so staleness is reproducible per the contract. Optional AI enrichment is a separately flagged, clearly labelled section per the contract's "Optional enrichment" rule and this repository's existing "AI outputs are proposals" cross-phase invariant (`PHASE-REVIEW.md:113`) â€” Phase 4 (AI Runtime) is not a Phase 3 dependency, so enrichment ships behind a feature flag that is off by default until Phase 4 exists to serve it, matching `ATTENTION-MODEL.md`'s and this repo's established "deterministic core, AI optional and gated" pattern from every prior phase.

## Frontend approach

New `frontend/src/features/attention/` mirroring `features/{tasks,notes,risks,knowledge}`: extend the existing attention surface (Phase 1 already has one â€” locate and confirm during Task 1, do not assume a green field) with waiting/risk-review/meeting groupings per `UX-STATES.md`'s "Attention queue" section, plus new `Planner.tsx` (Slice 5-6, timeline and accessible list views, drag-and-drop with keyboard equivalents) and `MeetingPrep.tsx` (Slice 7, facts/questions/suggestions visually separated). No new charting/scheduling UI library â€” `UX-STATES.md` requires an accessible list view as the primary representation alongside any timeline view, matching Phase 2's "no graph-visualization dependency" precedent for the same reason (keeps this slice inside RFC-005's existing frontend baseline).

## Approval decision gates (per `docs/phases/PHASE-REVIEW.md:128`)

Three decisions the repository owner must resolve before Phase 3 contracts move to Approved for Implementation. This document proposes a concrete starting answer for each so the decision is a review, not a blank page:

1. **Attention policy weights/caps.** Proposed: policy v1 = Phase 1's exact current weights (Open decision 1), extended additively with the new factors above; any future reweighting goes through a policy version bump with the before/after benchmark discipline `ATTENTION-MODEL.md`'s Evaluation section already requires.
2. **Critical-item definition.** Proposed, for the dogfood "no missed critical item" exit criterion: an item is critical if it is overdue, due/reviewable within 48 hours, or blocking a dependent item, matching the existing `overdue`/`due_48h`/`review_due_soon`/`review_overdue` factor codes `attention.py` already computes today â€” no new concept, just naming the existing high-weight factors as the critical set.
3. **Dogfood success thresholds.** Proposed, mirroring Phase 2's retrieval-benchmark precedent of naming a concrete numeric floor rather than leaving it qualitative: zero missed critical items across the two-week window, â‰Ą80% top-five usefulness rating, plan acceptance rate â‰Ą60% (below that, the planner is proposing infeasible plans), false-urgency rate <10%.
