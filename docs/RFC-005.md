# RFC-005 — Approved Technology Registry

## Purpose

This RFC is the canonical technology allowlist for Executive Command Center. No engineer or AI coding agent may introduce a framework, library, database, protocol, runtime or infrastructure component unless it is listed here for the active phase or approved through an ADR.

## Interpretation rules

1. **Required now** means the technology is part of the current Phase 0 implementation baseline.
2. **Approved later** means it may be used only when a future phase specification activates it.
3. **Evaluation only** means it is not approved for implementation.
4. Exact direct-dependency versions are recorded below and in lockfiles.
5. Transitive dependencies are controlled exclusively through `uv.lock` and `pnpm-lock.yaml`.
6. Floating versions such as `latest`, `3.x`, `5.x`, `^` and `~` are prohibited in manifests, container tags and CI actions.

## Phase 0 architecture baseline

```text
Browser
  ↓
React web application
  ↓
FastAPI modular monolith
  ↓
PostgreSQL 18
  ├── relational domain data
  ├── PKOS nodes, edges and provenance
  ├── durable event outbox/inbox
  └── session and workspace state
```

Phase 0 deliberately excludes Neo4j, Qdrant, Redis, NATS, Kafka, Kubernetes and cloud-managed services.

## Phase 2 activated technologies

Activated by this v1.2.0 amendment, per the "Retrieval benchmark and ADR" requirement above and ADR-0011. Unlike the Phase 0 runtime baseline, these are scoped to Phase 2's knowledge-platform hybrid retrieval (Task 7) specifically, not a Phase 0 dependency.

| Capability | Technology | Exact baseline | Status |
|---|---|---:|---|
| Vector storage and ANN search | pgvector (PostgreSQL extension) | via `pgvector/pgvector:pg18` image | Required for Phase 2 Task 7 |
| Vector storage and ANN search | pgvector (Python client, `pgvector` package) | 0.5.0 | Required for Phase 2 Task 7 |
| Local embedding generation | sentence-transformers | 5.6.0 | Required for Phase 2 Task 7 |
| Local embedding generation | Model: `sentence-transformers/all-MiniLM-L6-v2` | 384 dimensions | Required for Phase 2 Task 7 |

`pgvector/pgvector:pg18` replaces the plain `postgres:18.0` image everywhere a Phase 2-capable database is provisioned (`docker-compose.yml`, `.github/workflows/ci.yml`, `.github/workflows/phase1-acceptance.yml`) -- same PostgreSQL 18 core, with the vector extension's shared library pre-built in. This is a Docker-tag pin, not an extension-version pin, consistent with how the plain `postgres:18.0` tag itself doesn't separately pin PostgreSQL's internal package versions.

`sentence-transformers` pulls in `torch` as a transitive dependency (not independently pinned here; controlled via `uv.lock` per this document's Interpretation rule 5). This is the single largest dependency addition to date -- see ADR-0011's Consequences for the accepted tradeoff (genuine local-first inference vs. dependency footprint) and the alternatives considered (an external embedding API, rejected as contradicting ADR-0002's local-first principle).

Both technologies are off by the runtime `Settings.embeddings_enabled` flag (default `false`); their presence in the dependency manifest does not mean every deployment pays their runtime cost.

## Phase 4 activated technologies

Activated by this v1.3.0 amendment, per the "AI-runtime phase specification and ADR review" requirement above and ADR-0012. Scoped to Phase 4's first AI Runtime activation specifically (one local model, no remote provider, two read-only tools -- `docs/superpowers/specs/2026-07-23-phase-4-ai-runtime-design.md`), not a Phase 0 dependency.

| Capability | Technology | Exact baseline | Status |
|---|---|---:|---|
| Local model inference (server) | Ollama | via the official `ollama/ollama` container image, pinned by digest at CI-provisioning time (design doc's Test strategy section) | Required for Phase 4's AI Runtime activation |
| Local model inference (client) | `ollama` (Python package) | 0.6.2 | Required for Phase 4's AI Runtime activation |
| First registered local model | `qwen2.5:1.5b-instruct-q4_K_M` (Qwen2.5, 1.5B parameters, Q4_K_M quantization, Apache 2.0) | served through Ollama | Required for Phase 4's AI Runtime activation |

Ollama runs as a separate local process/service (already anticipated by `docs/architecture/chapter-02b-runtime.md`'s Docker Compose diagrams), not an in-process Python dependency -- unlike Phase 2's `sentence-transformers`/`torch` addition, the `ollama` Python client itself is a small pure-Python HTTP wrapper with no native-build or musl/Alpine packaging concern of its own (see ADR-0012's Consequences).

No remote model provider is approved by this amendment. `model_definitions` (`docs/phases/phase-004/DATA-MODEL.md`) contains exactly one row -- the Ollama model above, `deployment=local` -- eligible for every data class this activation defines; every data class is local-only-eligible and zero are remote-eligible in this first activation (ADR-0012's Decision 7 equivalent, "Data class policy" in the design doc). Adding a remote provider requires its own future RFC-005 amendment and ADR, not implied by this one.

This sandbox's environment cannot reach `ollama.com` or run the Ollama server binary (confirmed 403 from the environment's egress proxy); only the `ollama` Python client package (PyPI-installable) and contract-level tests against a mocked HTTP client were exercised while producing this amendment -- real end-to-end verification against a live Ollama service requires a dedicated CI job with genuine network access, mirroring the `embeddings-benchmark` job's role for Phase 2's `torch`-on-Alpine constraint (see ADR-0012's Risks and the design doc's Test strategy section).

## API and contract standards

- REST over HTTP
- OpenAPI 3.1
- JSON payloads
- UUID identifiers
- UTC timestamps in RFC 3339 form
- typed command, query and event contracts
- no GraphQL or gRPC in Phase 0

## Authentication baseline

Phase 0 uses:

- one local owner account per installation
- Argon2id password hashing
- server-side opaque sessions persisted in PostgreSQL
- `HttpOnly`, `Secure` outside local HTTP development, and `SameSite=Lax` cookies
- CSRF token validation for state-changing browser requests
- 12-hour idle timeout and 7-day absolute session lifetime
- session rotation after authentication and privilege-sensitive operations

JWT is not approved for local browser sessions. OAuth2 is reserved for external connector authorization and future multi-user identity.

## Observability baseline

Phase 0 uses structured JSON logs and application-native metrics endpoints. OpenTelemetry, Prometheus and Grafana are approved later and are not required local services in Phase 0.

Required log fields:

- timestamp
- level
- service
- domain
- workspace_id where applicable
- correlation_id
- causation_id where applicable
- request_id
- event_id where applicable
- message

Secrets, credentials, full prompt context and sensitive source content MUST NOT be logged.

## Version and lock policy

- Direct dependencies use exact equality pins.
- Python uses `requires-python == 3.14.*` plus a checked-in `.python-version` containing `3.14.5`.
- Node uses `.node-version` containing `22.17.0` and an exact `packageManager` field for pnpm.
- Docker images use immutable version tags and SHOULD record image digests after initial resolution.
- GitHub Actions MUST use immutable commit SHAs, with the human-readable release in a comment.
- `uv.lock` and `pnpm-lock.yaml` are mandatory and committed.
- Lockfile changes require CI, vulnerability scans and review.

## Upgrade policy

Patch and minor upgrades require:

1. compatibility tests
2. security scan
3. lockfile diff review
4. changelog review
5. rollback instructions

Major runtime, framework, database or infrastructure upgrades require an ADR.

## Prohibited in Phase 0

- floating dependency versions
- unpinned Docker images
- Neo4j
- Qdrant
- Redis
- Kafka
- NATS
- Temporal
- Kubernetes
- cloud-only dependencies
- JWT browser sessions
- no-auth development mode
- LangChain, LangGraph, CrewAI or AutoGen
- MongoDB, Firebase, Django, Flask, Express or Electron

## Fitness functions

- No dependency exists outside this registry or an accepted ADR.
- No floating version exists in manifests, workflows or image tags.
- Phase 0 Compose starts PostgreSQL only as shared infrastructure.
- Lockfiles reproduce on macOS and Linux.
- CI emits an SBOM and runs secret and vulnerability scans.
- Authentication tests prove cookie flags, CSRF protection, expiration and session rotation.
- No future-phase technology is required for local startup.

## Approval

RFC-005 version 1.1.0 is the approved technology baseline for Phase 0. Any deviation requires an ADR or an explicit versioned amendment to this RFC.
