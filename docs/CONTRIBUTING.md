# Contributing to Executive Command Center

Welcome.

Executive Command Center (ECC) is built differently from most software projects.

This repository is **Specification Driven**.

Architecture is defined before implementation.

Every contribution—whether written by a human or generated by AI—must preserve the architectural integrity of the platform.

Please read this document before contributing.

---

# Project Philosophy

Executive Command Center exists to become a Local-First Executive Operating System.

We optimize for

- Executive Attention
- Long-term Maintainability
- Explainability
- Architectural Consistency

We do **not** optimize for:

- Shipping features quickly
- Clever implementations
- Framework experimentation

---

# Read This First

Before writing code, read these documents in order.

1. SPEC-000 — Constitution
2. RFC-001 — Product Definition
3. RFC-002 — Engineering Philosophy
4. RFC-003 — Design Principles
5. RFC-004 — System Architecture
6. RFC-005 — Technology Registry
7. STD-001 — Repository Standards

Do not begin implementation until you understand them.

The repository follows

```

```

# Contribution Workflow

Every contribution follows the same lifecycle.

```
Issue

↓

Specification

↓

Architecture Review

↓

Implementation

↓

Testing

↓

Documentation

↓

Review

↓

```

No shortcuts.

---

# Before Writing Code

Verify

- Requirement exists
- Phase exists
- Technology approved
- Architecture understood
- Acceptance tests defined

If anything is unclear,

stop.

Create a Specification Change Request.

---

# Coding Standards

All code must comply with

STD-001

Repository Standards.

Highlights

- Small modules
- Typed code
- Tests required
- Documentation required
- Logging required
- No TODOs
- No dead code

---

# AI Contributions

AI-generated code is welcome.

However AI must never

- invent APIs
- invent requirements
- invent technologies
- invent architecture

Every AI contribution receives the same review as human code.

---

# Pull Requests

Every PR must include

- Problem Statement
- Solution
- RFC References
- FR References
- Tests
- Documentation Updates
- Architecture Impact
- Rollback Plan

---

# Commit Messages

Use Conventional Commits.

Examples

```
feat(attention): implement priority engine

fix(memory): resolve entity merge

docs(rfc004): update PKOS architecture

refactor(ai): simplify model router
```

---

# Branch Strategy

```
main

develop

feature/*

fix/*

docs/*
```

---

# Code Reviews

Every review verifies

- Correctness
- Simplicity
- Tests
- Documentation
- Security
- Architecture
- Traceability

Architecture takes precedence over implementation style.

---

# Documentation Rule

If code changes behaviour,

documentation changes in the same PR.

Specification drift is prohibited.

---

# Testing

Every feature requires

- Unit Tests
- Integration Tests
- Acceptance Tests

AI features additionally require

- Golden Dataset
- Regression Evaluation

---

# Questions

If the specification is unclear,

do not guess.

Create a Specification Change Request.

---

# Final Rule

Leave the repository better than you found it.

Small improvements accumulate into great systems.
