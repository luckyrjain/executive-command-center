# RFC-002 — Engineering Philosophy

> **Purpose**
>
> This RFC defines the engineering philosophy that governs every technical decision in Executive Command Center.
>
> Unlike architecture, this document answers **why** we build software the way we do.

---

# Executive Summary

Engineering organizations become inconsistent when every engineer optimizes for a different goal.

Some optimize for speed.

Some optimize for elegance.

Some optimize for flexibility.

Some optimize for abstraction.

ECC optimizes for something else:

> **Long-term maintainability while maximizing AI-assisted development.**

Every engineering decision should make the system easier to understand, easier to modify, and easier for both humans and AI to reason about.

This document is the constitution of engineering decisions.

---

# Philosophy

ECC is built on ten permanent engineering beliefs.

Every architectural decision should reinforce these beliefs.

---

# EP-001

## Specification Before Code

Code is not the source of truth.

The specification is.

The development lifecycle SHALL always follow this order.

```
Understand

↓

Specify

↓

Review

↓

Implement

↓

Test

↓

Deploy
```

Engineering work MUST begin by updating the specification.

No behavior-changing pull request may exist without a corresponding specification update.

---

# EP-002

## Simplicity Wins

Complexity compounds.

Every additional abstraction has a maintenance cost.

When multiple solutions satisfy the same requirement,

the simplest solution SHALL be preferred.

Questions every engineer should ask:

- Can this be deleted?
- Can this be smaller?
- Can this be simpler?
- Can this reuse existing functionality?

The easiest software to maintain is software that does not exist.

---

# EP-003

## Delete Before Add

Adding code is easy.

Removing code is difficult.

Before introducing new systems,

engineers MUST evaluate whether existing systems can solve the problem.

Features that increase complexity without reducing cognitive load SHOULD NOT be implemented.

---

# EP-004

## AI is a Junior Engineer

AI accelerates engineering.

AI does not replace engineering judgment.

Every AI-generated change requires:

- human review
- architectural validation
- automated testing
- specification traceability

AI may generate code.

Humans remain responsible for correctness.

---

# EP-005

## Local First

Local execution is the default.

Cloud execution is optional.

Reasons:

- privacy
- ownership
- lower latency
- resilience
- predictable cost

Cloud services may enhance capability.

They SHALL NOT become mandatory for core workflows.

---

# EP-006

## Explainability Over Intelligence

An explainable recommendation is more valuable than an opaque recommendation.

Every recommendation generated by ECC must answer:

- Why?
- Based on what?
- Which evidence?
- What confidence?

If the answer cannot be explained,

the recommendation SHOULD NOT be shown.

---

# EP-007

## Human Authority

AI recommends.

Humans decide.

AI SHALL NEVER silently:

- send email
- delete information
- modify data
- execute workflows
- approve requests

without explicit approval.

---

# EP-008

## Permanent Memory

Knowledge is cumulative.

Information should never be intentionally discarded.

ECC remembers.

Users may archive.

Users may delete.

The system itself should prefer preservation over deletion.

Summaries are regenerated.

Memory persists.

---

# EP-009

## Modular Architecture

Every subsystem should have one clear responsibility.

Subsystems communicate through contracts.

Not shared implementation.

Preferred dependency direction:

```
UI

↓

Application

↓

Domain

↓

Infrastructure
```

Never the reverse.

---

# EP-010

## Evolution Over Perfection

The architecture will evolve.

Perfection is not the goal.

Stability is.

Engineers should choose solutions that can evolve safely rather than attempting to predict every future requirement.

---

# AI Engineering Philosophy

ECC is intentionally designed for AI-assisted development.

Therefore the codebase must optimize for machine understanding as much as human understanding.

---

---

## AI Rule 2

---

## AI Rule 3

---

## AI Rule 4

Deterministic APIs.

---

## AI Rule 5

Self-documenting interfaces.

---

## AI Rule 6

Stable contracts.

---

## AI Rule 7

---

# Stop and Ask Protocol

This protocol is mandatory for both humans and AI coding agents.

Implementation MUST stop whenever:

- requirements are ambiguous
- specifications conflict
- requirements are missing
- security implications are unclear
- architectural ownership is uncertain

Instead of guessing,

the engineer MUST create a Specification Change Request.

Guessing is considered an engineering defect.

Engineering decisions follow this hierarchy.

```
RFC

```

Lower-level documents may refine higher-level documents.

# Engineering Principles

Every pull request should improve at least one of these.

## Readability

Code is read more often than written.

---

## Testability

Behavior should be independently testable.

---

## Replaceability

Dependencies should be replaceable.

---

## Observability

Failures should be visible.

---

## Reliability

Correctness before optimization.

---

## Maintainability

Future engineers should understand the system quickly.

---

## Predictability

The system should behave consistently.

---

## Incrementality

Large rewrites are discouraged.

Small continuous improvements are preferred.

---

# Architectural Fitness Functions

These are permanent architectural constraints.

## AFF-001

All LLM calls MUST pass through the Model Router.

---

## AFF-002

Connectors MUST NOT communicate directly with databases.

---

## AFF-003

Agents communicate through contracts.

---

## AFF-004

Every AI action is auditable.

---

## AFF-005

Every externally visible behavior traces back to an FR ID.

---

## AFF-006

Business logic SHALL NOT exist inside UI components.

---

## AFF-007

Secrets SHALL NEVER appear in source code.

---

## AFF-008

Prompt versions SHALL be version controlled.

---

## AFF-009

Every subsystem SHALL expose health information.

---

## AFF-010

Every production change SHALL be reversible.

---

# Technical Debt Philosophy

Technical debt is not forbidden.

Unmanaged technical debt is.

Every intentional shortcut MUST record:

- reason
- impact
- owner
- review date
- removal plan

Invisible technical debt is unacceptable.

---

# Engineering Metrics

The engineering organization should continuously improve:

- deployment frequency
- change failure rate
- lead time
- MTTR
- test coverage
- documentation coverage
- AI implementation success rate
- architecture violations
- dependency health

---

# Definition of Good Software

Good software is:

- understandable
- observable
- testable
- replaceable
- explainable
- resilient
- documented

Good software is not necessarily clever.

---

# Anti-Patterns

The following behaviors are discouraged.

- Premature abstraction
- Hidden side effects
- Magic configuration
- Unbounded dependencies
- Copy-paste architecture
- Tight coupling
- Silent failures
- Business logic inside prompts
- Business logic inside UI
- Circular dependencies

---

# Engineering Oath

Every contributor should leave the repository in a better state than they found it.

Before merging code, ask:

- Is it simpler?
- Is it easier to understand?
- Is it easier to test?
- Is it easier to change?
- Does it align with the specification?

If the answer is no,

reconsider the implementation.

---

# Acceptance Criteria

This RFC is considered adopted when:

- All future RFCs align with these principles.
- Repository standards enforce these rules.
- Architecture decisions reference this philosophy.
- AI coding guidelines inherit from this RFC.
- New contributors can understand the engineering values without tribal knowledge.

---

# Summary

RFC-002 establishes the engineering philosophy for Executive Command Center.

It defines:

- how decisions are made
- how software evolves
- how AI participates in development
- how architecture remains maintainable
- how engineering quality is measured

Every future engineering decision should be explainable through the principles established in this RFC.
