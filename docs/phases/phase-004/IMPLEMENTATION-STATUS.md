Chose `qwen2.5:3b-instruct-q4_K_M` over the design doc's originally-named `llama3.2:3b-instruct-q4_K_M` (repository owner's explicit call): same license family (Apache 2.0) and model family as the first model, avoiding a second license to track, while still giving a real ~2x parameter-count delta between candidates.

**A real, second-model-caused correctness gap found and fixed while implementing this.** `evaluation.py`'s `run_evaluation(model_id=...)` was documented as an "assertion, not an override" -- verifying the requested model is registered and eligible, then letting `execute_run` route normally -- but `execute_run` has no way to pin a specific candidate, and nothing checked that the model it actually routed to matched the one requested. With a single registered model this was moot (only one possible outcome); with two, `route()` could silently pick a different candidate than requested, and `EvaluationRun.model_id` (stamped from the request parameter, not from what actually ran) would misattribute the results. Fixed: `run_evaluation` now raises `EvaluationConfigError("unexpected_model_routed", ...)` immediately if any example's routed model differs from the requested one, matching the module's own "fail loud, never silently score a run against the wrong model" contract. Covered by `tests/test_ai_runtime_evaluation_postgres.py::test_run_evaluation_requested_model_not_actually_routed_raises_config_error`, which relies on real deterministic tie-break behavior (both models start with zero observed history in a fresh test database) rather than a contrived mock.

**Tests.** `tests/test_ai_runtime_routing_postgres.py`: five existing "exactly one row/model" assertions updated to two; a new test, `test_refresh_cache_and_route_pick_correctly_among_two_real_seeded_models`, proves the real migration-seeded registry round-trips correctly through `refresh_cache` (previously untested with any real data) into a correct routing decision -- the missing link between the existing synthetic-fixture preference tests (already exhaustive against fake multi-candidate data) and the real database. `tests/test_ai_runtime_evaluation_postgres.py`: the new `unexpected_model_routed` test above. `tests/test_ai_runtime_evaluation_live_ollama.py`: a new single-item smoke test, `test_second_registered_model_produces_a_valid_completed_run_against_real_ollama`, proving the second model is genuinely invokable against a real Ollama server (not just present in the registry) -- deliberately not a full 20-example evaluation-floor run against it, which is a separate promotion decision from "this model works at all"; the first model is temporarily marked `disabled` for this test's duration (restored in a `finally` block) so eligibility, not a preference tie-break, determines the routed candidate. `.github/workflows/ollama-evaluation.yml` updated to pull both model tags.

### Task 8 evidence -- Reflection Engine (first slice) on `attention.explain_item`

The repository owner's re-raised deferred-scope item -- "Agent Runtime, multi-agent coordination, Reflection Engine" (design doc's Decision 8 deferred-scope line) -- narrowed to its smallest concrete first piece, per an explicit follow-up decision: one additional, optional, bounded, fail-open model call on `attention.explain_item`, not the design doc Chapter 3's fuller multi-agent Coordinator/Agent-Runtime architecture.

**Backend.** `backend/ecc/domains/ai_runtime/runtime.py`: `_reflect_on_answer` (new), called from `execute_run` after grounding passes and before the run is persisted `completed`. Reuses the existing `call_model` closure verbatim (inherits `RunGuard`'s total-wall-clock budget check with zero new mechanism) and the existing `_render_factors_block`/untrusted-data-delimiter convention (refactored into a shared `_wrap_untrusted_data` helper, reused by a new `_render_prior_answer_block` for the model's own prior answer). Every failure mode (timeout, cancelled, provider error, total-wall-clock budget exceeded, output-token budget exceeded, tool-call-shaped response, schema-invalid reflection response, a proposed revision that itself fails `validate_output`/`check_explain_item_grounding`) is fail-open: the run always completes with the original, already-validated-and-grounded answer. Only a revision that independently passes the exact same validation/grounding checks the original did can replace it. Deliberately does not feed `_breaker_for(...)`'s circuit breaker -- a reflection-layer failure must not degrade the primary task's routing eligibility for that model.

**Review-driven fix.** A 4-dimension parallel review (security, performance, adversarial test-verification, PR hygiene) of this PR found one real, minor defense-in-depth gap: `execute_run`'s primary call re-checks its `eval_count` against `budget.max_output_tokens` via `check_output_token_budget` (a fallback safety net for a model that ignores `num_predict`) but the reflection call's own `eval_count` was never checked the same way. Fixed by threading `budget` into `_reflect_on_answer` and re-running the same check, fail-open (`outcome="output_budget_exceeded"`) on the reflection call specifically, distinct from the pre-existing `outcome="budget_exceeded"` (total wall-clock). Not a vulnerability on its own (`cost` is always `0.0` in this activation), but closes the same gap the primary path already guards against.

`backend/ecc/domains/ai_runtime/validator.py`: `ExplainItemReflection` (new, `{approved, revised_explanation_text, revised_cited_factor_codes}`, `extra="forbid"`) -- deliberately no word-count validator of its own; a proposed revision is re-validated as a fresh `ExplainItemOutput`, so the 60-word rule keeps exactly one owner.

`backend/ecc/domains/ai_runtime/budgets.py`: `reflection_enabled(policy)` -- reads `routing_policies.constraints.reflection_enabled` (default `False`), the same JSONB column `RunBudget.from_policy` already reads its five budget numbers from.

**Migration.** `backend/migrations/versions/0033_phase4_reflection.py`: an in-place `jsonb || jsonb` merge onto the existing active `attention.explain_item` policy row's `constraints` (seeding `reflection_enabled: false`, following `0032_phase4_second_model.py`'s in-place-update precedent), plus a new, separate `prompt_versions` row (`prompt_id='attention.explain_item.reflect.v1'`, following `0029_phase4_prompt_tool_versions.py`'s seeding pattern). A real bug was found and fixed while authoring this migration: `sa.cast(<python str>, JSONB())` double-encodes the string through the JSONB bind processor, turning an intended object merge into a two-element array (`[original_object, "quoted json string"]`) -- confirmed against a real local Postgres instance, not a hypothetical. Fixed by passing a Python **dict** via `sa.literal(..., type_=JSONB())` instead, so the bind processor's single `json.dumps` produces a real object. Migration verified reversible (`upgrade` -> `downgrade -1` -> `upgrade` clean against a local Postgres instance, including re-verifying the merge/removal produces the exact original `constraints` object back).

**Tests.** `tests/test_ai_runtime_runtime_postgres.py`: 7 tests exercising `execute_run` end-to-end against a mocked transport -- disabled-by-default skips the call; approved keeps the original; a valid+grounded revision replaces the output (asserting the full original answer, not just its cited codes, is preserved on discard); a revision that fails grounding is discarded with the original kept; a schema-invalid reflection response is never repair-retried (proven via an actual HTTP call-count assertion, not just the final step-kind list); a tool-call-shaped reflection response is rejected, never dispatched (reusing Step 1/5's allowlist-injection test pattern); a reflection-layer provider error is skipped, run still completes. Plus 7 tests exercising `_reflect_on_answer` directly (timeout, cancelled, total-wall-clock budget exceeded, output-token budget exceeded, an over-60-word revision failing `ExplainItemOutput` re-validation, `revised_cited_factor_codes=None` falling back to the original codes, `approved=False` with no revision text keeping the original) -- covering fail-open branches impractical to force through a real timing/budget race via the full orchestration path. `tests/test_ai_runtime_validation_postgres.py`: 5 tests for `ExplainItemReflection`'s schema shape, including the explicit "an over-60-word revision validates fine as a Reflection object, only rejected at the `ExplainItemOutput` re-validation step" boundary. `tests/test_ai_runtime_budgets_postgres.py`: 4 tests for `reflection_enabled` (seeded-row read, explicit true/false, defaults false when the key is absent). `tests/test_ai_runtime_evaluation_live_ollama.py`: one smoke test proving the reflection call is genuinely invokable end-to-end against a real Ollama server (not run as part of the promotion-floor check -- see `EVALUATION-CONTRACT.md`'s "Reflection Engine (first slice) and this evaluation's floors" section for why).

These test counts and the token-budget fix above reflect a post-open-PR review pass (4 parallel review agents: security, performance, adversarial test-verification, PR hygiene) -- security, performance, and PR-hygiene dimensions found no issues; the adversarial test-verification pass found two weak assertions (now strengthened, above) and six untested fail-open branches (now covered, above).

**Docs.** `DATA-MODEL.md`, `MODEL-ROUTING-CONTRACT.md`, `API-SCHEMAS.md`, `EVALUATION-CONTRACT.md` updated for the new `constraints` key, `prompt_versions` row, `ai_run_steps` step shape, no-new-routing-decision clarification, and the deliberate reflection-disabled-during-floor-check tradeoff. ## What remains before Phase 4 itself can exit

- `.github/workflows/ollama-evaluation.yml`'s first real runs (plural) happened as part of PR #38's review cycle, against a real `ollama/ollama` container and `qwen2.5:1.5b-instruct-q4_K_M`. The first run found `EVALUATION-CONTRACT.md`'s floors genuinely unmet (55% schema validity, 50% grounding vs. 100%/100% required) -- an unverified floor was treated as unresolved, not passing by assumption (design doc's Test strategy section), exactly as this document originally committed to. Two real, evidence-driven code fixes followed:
  - `validator.py`'s `validate_output` now strips a well-formed leading/trailing markdown code fence before strict JSON parsing -- small instruct-tuned models commonly wrap JSON output in ` ```json ... ``` ` out of chat-formatting habit even when told not to. This alone raised schema validity to 90-100% and grounding to 70-85% across subsequent runs.
  - `runtime.py`'s `_render_factors_block` (the prompt-rendering function, not the immutable Decision-4 template) was reformatted after two further live runs showed the model repeatedly citing a factor's `source_field` value instead of its `code` value in `cited_factor_codes` (e.g. citing `"created_at"` instead of `"recently_created"`) -- a systematic confusion, not random hallucination, traced to the old rendering format putting `code` and `source_field` on equal visual footing. This raised grounding to 90% with zero grounding-related failures in the most recent run.
