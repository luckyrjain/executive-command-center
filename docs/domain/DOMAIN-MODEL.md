# Canonical Domain Model

## Purpose

This document defines the business objects and ownership boundaries used across ECC. It is normative. Implementations may add internal fields but may not redefine these concepts without an ADR.

## Universal rules

Every entity has:

- `id`: immutable UUID generated by ECC
- `workspace_id`: isolation boundary
- `created_at`, `updated_at`: UTC timestamps
- `version`: optimistic concurrency number
- `source_refs[]`: zero or more external identities
- `provenance`: origin and evidence metadata where the entity is derived

External identifiers are never used as ECC primary keys. Deletion is soft by default and records `deleted_at` plus provenance.

## Ownership map

| Domain | Owns |
|---|---|
| Identity | Workspace, User, Person, Organization |
| Planning | Task, Goal, Project, CalendarEvent, Reminder |
| Communication | Conversation, Message, EmailThread, Commitment |
| Executive Intelligence | Risk, Brief, Recommendation, AttentionItem |
| Knowledge Platform | KnowledgeItem, Document, Decision, Relationship, Evidence |
| AI Platform | PromptDefinition, ModelExecution, EvaluationResult, AgentRun |
| Integration Platform | ConnectorAccount, SyncCursor, SourceRecord |

A domain may reference another domain's entity ID but may not directly mutate the other domain's storage.

## Core entities

### Workspace

The top-level data and authorization boundary.

Lifecycle: `active -> suspended -> archived`.

### User

An authenticated actor within a workspace. Phase 0 supports one owner user but preserves multi-user identifiers.

### Person

A real human known to ECC. A Person may or may not be a User. Identity resolution may merge duplicate Person records while preserving merge history.

### Organization

A company, team, family or external group associated with people, projects and source systems.

### Project

A bounded outcome with owner, status, dates, goals, risks and related work.

Lifecycle: `proposed -> active -> blocked|on_hold -> completed|cancelled -> archived`.

### Goal

A desired measurable outcome. Goals may contain projects and success measures.

### Task

A discrete unit of work with an accountable owner, status, priority, due date and source.

Lifecycle: `captured -> planned -> in_progress -> blocked -> completed|cancelled`.

### Commitment

A promise made by or to a person. Commitments differ from tasks because they preserve the social contract, the parties and supporting evidence.

Lifecycle: `detected -> confirmed -> active -> fulfilled|broken|cancelled`.

AI-detected commitments remain `detected` until human confirmation or deterministic source rules validate them.

### CalendarEvent

A scheduled time interval imported from or created for a calendar. The external calendar remains authoritative unless ECC creates the event through an approved action.

### Meeting

A semantic view of a CalendarEvent with participants, agenda, preparation material, notes, decisions and follow-ups. A meeting may exist without a calendar event during planning.

### Conversation

A logical thread across one communication channel, containing ordered messages and participants.

### Message

An immutable communication artifact. Email is represented as a Message with email-specific metadata; original content is preserved as Evidence.

### Document

A source artifact or user-authored content item with version, checksum, media type and storage location.

### Decision

A recorded choice with context, alternatives, rationale, decision maker, effective date and review conditions.

Lifecycle: `proposed -> accepted -> superseded|reversed|expired`.

### KnowledgeItem

A normalized fact, concept, summary or assertion held by PKOS. Derived items carry confidence, provenance and validity dates.

### Relationship

A typed, directed connection between two entities, including provenance, confidence and temporal validity.

### Evidence

An immutable pointer to the source supporting a fact, decision, commitment or recommendation. Evidence includes source location, checksum, captured timestamp and excerpt boundaries.

### Risk

An uncertain condition with probability, impact, owner, mitigation, trigger and review date.

Lifecycle: `identified -> assessed -> monitoring|mitigating -> materialized|closed`.

### AttentionItem

A ranked item presented to the executive. It references the underlying entity and records priority factors, explanation, confidence and expiry.

### Recommendation

An AI- or rule-generated proposal. It cannot mutate domain state directly. It records evidence, explanation, confidence and required approval.

### Reminder

A request to surface an entity at a time or when a condition becomes true.

## Important relationships

```text
Person MEMBER_OF Organization
Person PARTICIPATES_IN Meeting
Person OWNS Task
Person MAKES Commitment
Commitment RELATES_TO Project
Meeting PRODUCES Decision
Meeting PRODUCES Task
Document SUPPORTS Decision
Evidence SUPPORTS KnowledgeItem
Project ADVANCES Goal
Risk THREATENS Project
AttentionItem HIGHLIGHTS any entity
KnowledgeItem ABOUT any entity
```

## Identity and merge rules

- Source identities map to ECC entities through `source_refs`.
- Automated entity resolution may propose merges but never silently merge high-impact entities.
- A merge preserves both original IDs as aliases and creates an auditable merge record.
- Unmerge must be possible while source evidence remains available.

## Source-of-truth rules

- External systems remain authoritative for imported messages, calendar events and source-control artifacts.
- ECC is authoritative for locally created tasks, commitments, decisions, risks, notes, relationships and recommendations.
- Derived knowledge is never treated as source evidence.

## Phase 0 freeze

The entities, ownership map and lifecycle states in this document are frozen for Phase 0. Changes require an ADR and updates to API contracts, event schemas and tests.
